สรุปภาพรวม

โดยรวมมีการเปลี่ยนแปลงจำนวน 12 รายการ (เพิ่ม 0 รายการ, ลบ 0 รายการ, แก้ไข 12 รายการ)

ส่วนที่ 1 – สรุปภาพรวมการเปลี่ยนแปลง
- ลด Response Time ของบริการให้คำปรึกษาออนไลน์จาก 4 ชม. เป็น 2 ชม. (Page 2)
- เพิ่มเงื่อนไขการจัดการ Critical Incident พร้อมกำหนด Resolution Time (Workaround) 4 ชม. และระบุช่องทางติดต่อฉุกเฉิน (Page 2)
- ขยายขอบเขตของ Data Engineer ให้ครอบคลุมการดูแลความปลอดภัย, การอัปเดต Patch/LLM, รายงานไตรมาส, และการอัปเดตระยะไกล (Page 2)
- เพิ่มรายละเอียดการสำรองและกู้คืนระบบ KM‑AI: Retention Docker 3 copy, Image 3 copy, Volume 30 วัน, Model 3 copy; กำหนด RTO 1 วัน (เดิม 4 ชม.) และ RPO 24 ชม. (Page 5)
- การแก้ไขเชิงอักษรและรูปแบบอื่น ๆ จำนวน 12 รายการไม่มีผลต่อความเสี่ยง (Page 1, 2, 4, 7)

ส่วนที่ 2 – ข้อเสนอแนะภาพรวม

1) มุมมองผู้ได้รับบริการ (ลูกค้า)
- ตรวจสอบให้ SLA ใหม่ (Response 2 ชม., Resolution 4 ชม.) ระบุชัดเจนในสัญญาและกำหนด Penalty หากไม่บรรลุ
- จัดทำแผนรับมือเหตุฉุกเฉินและกระบวนการส่งข้อมูล (log, รายละเอียดเหตุการณ์) ให้ผู้ให้บริการ
- ประเมินผลกระทบด้านต้นทุนจากการขยายขอบเขต Data Engineer และพิจารณาเงื่อนไขค่าใช้จ่ายเพิ่มเติม
- ยืนยันว่าระดับ Retention ของ Backup ตรงตามนโยบายภายใน และตรวจสอบความพร้อมของระบบแจ้งเตือนอีเมล (SMTP, SPF/DKIM/DMARC)
- พิจารณาความเหมาะสมของ RTO 1 วัน; หากต้องการฟื้นฟูภายใน 4 ชม. ควรเสริมกระบวนการ Backup แบบต่อเนื่องหรือเพิ่มจุดสำรอง

2) มุมมองผู้ให้บริการ (ผู้ขาย)
- ปรับโครงสร้างทีมสนับสนุนให้มีทรัพยากรเพียงพอ (เพิ่ม Engineer, กำหนด Shift 24 ชม.) เพื่อให้บรรลุ SLA 2 ชม.
- จัดทำ SOP การจัดการ Critical Incident รวมช่องทางติดต่อฉุกเฉินและขั้นตอน Workaround 4 ชม.
- พัฒนาและบังคับใช้กระบวนการ Change Management ครอบคลุม Patch, LLM, OS, และการทดสอบความปลอดภัยก่อนนำไปใช้
- สร้าง SOP Backup/Restore ตาม Retention ที่กำหนด, ทดสอบ DR อย่างน้อยปีละ 1 ครั้ง, และยืนยันว่า RTO 1 วันและ RPO 24 ชม. สามารถทำได้จริง
- จัดเตรียมระบบแจ้งเตือนอีเมลที่ปลอดภัย, เข้ารหัสข้อมูลสำรอง (at‑rest, in‑transit) และกำหนดสิทธิ์เข้าถึงอย่างเข้มงวด
- ปรับสัญญาให้รวมค่าใช้จ่ายเพิ่มเติม (Manday, License, เครื่องมือ) และกำหนด Penalty หากไม่บรรลุ SLA หรือขอบเขตบริการใหม่

ส่วนที่ 3 – ผู้มีส่วนได้ส่วนเสียและผลกระทบ

- ลูกค้า
• ต้องจัดสรรบุคลากรและกระบวนการภายในเพื่อรองรับ SLA 2 ชม. และการแจ้งเหตุฉุกเฉิน
• เสี่ยงต่อค่าใช้จ่ายเพิ่มจาก Data Engineer ขยายขอบเขตและอาจต้องจ่ายค่า Service‑Level Penalty
• เสี่ยงต่อการสูญเสียข้อมูลเมื่อ Volume ทำ Full Restore หากไม่มี Backup ต่อเนื่อง
• ต้องปรับนโยบายภายในให้สอดคล้องกับ RTO 1 วันและ RPO 24 ชม.

- ผู้ให้บริการ (Softnix / VSTECS)
• ต้องเพิ่มจำนวน Engineer, จัดตั้ง Shift 24 ชม., และฝึกอบรมทีมเพื่อให้ตอบสนอง SLA ใหม่
• ต้องขยายขอบเขตงานของ Data Engineer รวมความปลอดภัย, Patch, LLM, รายงานไตรมาส ซึ่งเพิ่มภาระและค่าใช้จ่าย
• ต้องพัฒนาระบบ Backup/Restore, ระบบแจ้งเตือนอีเมล, และทำการทดสอบ DR อย่างสม่ำเสมอเพื่อหลีกเลี่ยง Penalty
• ต้องเจรจาปรับสัญญาเพื่อสะท้อนค่าใช้จ่ายและความเสี่ยงที่เพิ่มขึ้นจากการขยายขอบเขตบริการ.

ฝึกอบรมที่เกี่ยวข้องกับการนำระบบSoftnix GenAI มาใช้งาน วัตถุประสงค์ของเอกสารนี้คือการระบุรายละเอียด

2.ขอบเขตการให้บริการ
การให้บริการภายใต้ SOW นี้ประกอบด้วย:
2.1การออกแบบและติดตั/งระบบ
2.1.1การออกแบบและกําหนดค่าระบบ
- วิเคราะห์ความต้องการของลูกค้าและออกแบบระบบ Softnix GenAI ให้เหมาะสม
-กําหนดค่าระบบตามแผนการออกแบบที่ได้รับการอนุมัติ
2.1.2 การตั้งค่า AI Agent Services
- ปรับแต่ง AI Agent Services เพื่อรองรับการใช้งานเฉพาะของลูกค้า โดยมีรายละเอียดดังนี้:
a) การปรับปรุงข้อมูล:
-เตรียมและปรับปรุงข้อมูลสําหรับAI Knowledgeจํานวนไม่เกิน10 ชุดข้อมูล
-ทําความสะอาดและจัดรูปแบบข้อมูลให้เหมาะสมสําหรับการใช้งานกับAI
b) การออกแบบ Prompt Instruction:
-สร้างและปรับแต่งคําสั่งและคําแนะนําสําหรับAI ให้เหมาะสมกับลักษณะการใช้งานของ
ลูกค้า
- ทดสอบและปรับปรุง Promptเพื่อให้ได้ผลลัพธ์ที่แม่นยําและตรงตามความต้องการ
c) การเลือกและตั้งค่า LLM:
- คัดเลือก Large Language Model (LLM) ที่เหมาะสมกับความต้องการของลูกค้า (รายการ
LLM ที่ลูกค้าใช้ได้จะขึ้นอยู่กับรุ่นของสินค้า)
- ปรับแต่งพารามิเตอร์ของ LLM เพื่อให้ได้ประสิทธิภาพสูงสุด
d) การเชื่อมต่อกับระบบภายนอก:
-ดําเนินการเชื่อมต่อกับLine Services (ลูกค้าเป็นผู้จัดหา Line Services)
- ทดสอบการเชื่อมต่อและแก้ไขปัญหาที่อาจเกิดขึ้น
e) การติดตั้ง Web Chat Application:
- ติดตั้ง Web Chat Application บนระบบของลูกค้า
- หรือ ผสานรวม Web Chat เข้ากับ Web Application ที่มีอยู่ของลูกค้า
-ทดสอบการทํางานและปรับแต่งตามความต้องการ
2.2การฝึกอบรม
2.2.1 การอบรมการใช้งานระบบ ระยะเวลา 1 วัน โดยครอบคลุมเนื้อหาดังต่อไปนี้
- ครอบคลุมหัวข้อ: การใช้งานพื้นฐาน, การจัดการผู้ใช้, การตั้งค่าระบบ, และการแก้ไขปัญหา
เบื้องต้น, หลักการของ Prompt Engineering, เทคนิคการเขียน Prompt ที่มีประสิทธิภาพ, การ
ปรับแต่ง Prompt เพื่อผลลัพธ์ที่ดีขึ้น
2.3บริการหลังการส่งมอบ
บริการหลังการส่งมอบจะมีให้ตลอดระยะเวลา Subscription โดยมีรายละเอียดดังนี้:
2.3.1การให้คําปรึกษาออนไลน์
- ให้คําปรึกษาเกี่ยวกับการใช้งานการแก้ไขปัญหา การปรับแก้ Promptและการกําหนดค่าระบบ
- บริการให้คําปรึกษาแบบ8x5 (8 ชั่วโมงต่อวัน, 5 วันต่อสัปดาห์)
- Response Time: ภายใน4ชั่วโมงหลังจากได้รับแจ้งปัญหา
- Resolution Time (Workaround) : ภายใน 24ชั่วโมงสําหรับปัญหาทั่วไป
2.3.2 บริการ Data EngineerสําหรับChange Request
• จัดสรร Data EngineerสําหรับChange Requestจํานวน5 Manday
Change Request คือคําขอให้มีการเปลี่ยนแปลงหรือปรับปรุงระบบที่ได้ติดตั้งและกําหนดค่าไว้
แล้ว โดยการเปลี่ยนแปลงนี้อยู่นอกเหนือจากขอบเขตงานเดิมที่ได้ตกลงกันไว้ในตอนเริ่มโครงการ
นอกเหนือจากบริการให้คําปรึกษาออนไลน์ได้ฟรี

การแก้ไขเหล่านี้เป็นเพียงการปรับปรุงการพิมพ์/การจัดรูปแบบ ไม่ได้ส่งผลต่อเนื้อหาเชิงกฎหมาย, การเงิน หรือการบริหารโครงการ ดังนั้น **ไม่พบความเสี่ยงที่มีนัยสำคัญจากการเปลี่ยนแปลงนี้**. 2. **การเปลี่ยนแปลงเชิงสาระสำคัญ** - **Response Time ของบริการให้คำปรึกษาออนไลน์** - เดิม: ภายใน **4 ชั่วโมง** หลังได้รับแจ้งปัญหา - ใหม่: ลดลงเป็น **2 ชั่วโมง** - ผลกระทบ: เพิ่มภาระการตอบสนองของผู้ให้บริการ ต้องมีการจัดสรรทรัพยากรเพิ่มเพื่อให้สอดคล้องกับ SLA ที่เข้มงวดกว่าเดิม. - **การเพิ่มเงื่อนไขและขั้นตอนการจัดการ Critical Incident** - เพิ่มรายการใหม่ว่า "กรณี Critical Incident ที่ทำให้เกิด Server down ที่ไม่ใช่เป็นปัญหาที่เกิดจาก Hardware หรือ Service Down User ใช้งานไม่ได้". - ระบุช่องทางการติดต่อ (หมายเลขโทรศัพท์) และกำหนด **Resolution Time (Workaround) ภายใน 4 ชั่วโมง** (โดยมีหมายเหตุ "เบิ้มแจ้งลค 1 วัน") รวมถึงระบุว่า "Service Down ใช้เวลา 4 ชั่วโมง". - ผลกระทบ: ขยายขอบเขตการให้บริการให้ครอบคลุมเหตุการณ์ระดับวิกฤติที่อาจส่งผลต่อการดำเนินธุรกิจของลูกค้าอย่างรุนแรง ต้องมีแผนการตอบสนองและทีมสำรองพร้อมทำงานตลอดเวลา. - **การขยายขอบเขตของบริการ Data Engineer สำหรับ Change Request** - เดิม: เพียงจัดสรร Data Engineer จำนวน **5 Manday** สำหรับ Change Request เท่านั้น. - ใหม่: ขยายให้ครอบคลุมหลายด้าน ได้แก่ * ปัญหาที่เกิดจากการทำงานผิดพลาดของซอฟต์แวร์แพลตฟอร์ม Softnix Gen AI และ OS Rocky Linux * ช่องโหว่ด้านความปลอดภัยของซอฟต์แวร์แพลตฟอร์ม, ระบบปฏิบัติการ, และ Software Library ที่เกี่ยวข้อง * บริการ Update Patch, New Feature สำหรับ Minor Version ของซอฟต์แวร์และ OS * บริการอัปเดตเวอร์ชัน LLMs (Local LLMs) รวมถึง OS Rocky Linux * ส่ง Report ทุก Quarter เกี่ยวกับการอัปเดต patch/software/security * การอัปเดตทำนอกเวลาทำการเท่านั้น โดยใช้ Remote update ที่ VSTECS จัดเตรียม - ผลกระทบ: ขอบเขตการทำงานของ Data Engineer เพิ่มจากการแก้ไขเฉพาะ Change Request ธรรมดา ไปเป็นการดูแลด้านความปลอดภัย, การอัปเดตซอฟต์แวร์, การรายงานประจำไตรมาส และการจัดการอัปเดตระยะไกล ซึ่งอาจต้องเพิ่มทรัพยากรบุคคล, ความเชี่ยวชาญพิเศษ, และกำหนดกระบวนการควบคุมการเปลี่ยนแปลง (Change Management) ที่เข้มงวดยิ่งขึ้น. 3. **ผลกระทบต่อด้านกฎหมาย, การเงินและการบริหารโครงการ** - ข้อความใหม่ได้เพิ่มการจัดแบ่งเป็นหัวข้อย่อย 4.1, 4.2, 4.3 (และมีหัวข้อ 4.3 ซ้ำ) ทำให้โครงสร้างของข้อกำหนดชัดเจนขึ้น แต่การเพิ่มหัวข้อย่อยเองไม่เป็นความเสี่ยงสำคัญ 2. **การเพิ่มเนื้อหาใหม่เกี่ยวกับการสำรองและการกู้คืนระบบ (Backup & Restore)** - **4.2 Backup KM‑AI System** - ระบุขั้นตอนและระดับการสำรองข้อมูล (Retention) ได้แก่ Docker config 3 copy, Docker Image 3 copy, Docker Volume 30 วัน, LocalLLM Model 3 copy - แบ่งการสำรองเป็นแบบ Internal Backup (อ้างอิง Retention ด้านบน, มีสคริปต์ auto‑delete หรือเขียนทับเมื่อครบ 30 วัน) และ External Backup ที่ VSTECS ดูแลตาม Destination path ที่ Softnix แจ้งในเอกสาร Config summary system KM‑AI VST - เพิ่มการแจ้งเตือนทางอีเมลเมื่อ Backup สำเร็จหรือไม่สำเร็จ - **4.3 Restore KM‑AI System** (สองส่วนที่แยกกัน) - ส่วนแรกอธิบายกระบวนการ Restore จาก Internal Backup เป็นหลัก; ใช้ External Backup ก็ต่อเมื่อ Internal Backup ไม่สามารถใช้งานได้ - ระบุว่า Restore ของ Data (Volume) จะเป็น Full Restore ทำให้ข้อมูลที่เพิ่มหลังการ Backup จะหายทั้งหมด ส่วนการ Restore ของ compose, image, LLM จะไม่กระทบ Data Volume - กำหนดบทบาทของ Softnix และ VSTECS ในการจัดหาและติดตั้ง Nvidia driver หากมีการเปลี่ยนการ์ดกราฟิก - กำหนด Recovery Time Objective (RTO) – เดิมระบุ 4 ชั่วโมงของ Critical incident; แก้เป็น 1 วัน (OS 4 ชั่วโมงพอได้) หลังจาก HPE เปลี่ยนฮาร์ดแวร์ - กำหนด Recovery Point Objective (RPO) – 24 ชั่วโมง - ส่วนที่สองของ 4.3 (ซ้ำหัวข้อ) เพิ่มข้อกำหนดการทำ Monitoring ระบบเมื่อเกิดปัญหา พร้อม Alert ส่งอีเมลโดย VSTECS จัดเตรียม SMTP Server เช่น การแจ้งเตือน CPU, Memory, Disk เกิน Threshold, การแจ้งผล Backup & Restore 3. **ผลกระทบต่อความเสี่ยงและภาระหน้าที่** - **ภาระหน้าที่ของลูกค้า**: ข้อกำหนดเดิมเกี่ยวกับการจัดเตรียมข้อมูล, สิทธิ์เข้าถึง, การจัดสรรบุคลากร, และการแจ้งปัญหา ยังคงอยู่โดยไม่มีการเปลี่ยนแปลงสาระสำคัญ - **ภาระหน้าที่ของผู้ให้บริการ (Softnix, VSTECS)**: เพิ่มบทบาทชัดเจนในการทำ Backup/Restore, การจัดเตรียม SMTP Server, การจัดหาและติดตั้ง Nvidia driver, และการกำหนด RTO/RPO ซึ่งอาจทำให้ต้องมีการจัดสรรทรัพยากรและกระบวนการเพิ่มเติม - **ความเสี่ยงด้านการดำเนินงาน**: - การกำหนด RTO ใหม่เป็น 1 วันอาจเพิ่มความเสี่ยงต่อการหยุดทำงานของระบบในกรณีเหตุฉุกเฉินที่ต้องการฟื้นฟูภายใน 4 ชั่วโมงเดิม - การทำ Full Restore ของ Volume ทำให้ข้อมูลที่เพิ่มหลังการสำรองจะหายทั้งหมด หากไม่มีการสำรองแบบต่อเนื่อง (incremental) จะเสี่ยงต่อการสูญเสียข้อมูลสำคัญ - การพึ่งพา External Backup ของ VSTECS ต้องมั่นใจว่า Destination path และการแจ้งเตือนทำงานได้อย่างต่อเนื่อง มิฉะนั้นอาจเกิดช่องโหว่ในการกู้คืนข้อมูล - **ความเสี่ยงด้านความปลอดภัย**: การเพิ่มการส่งอีเมลแจ้งเตือนต้องมีการจัดการ SMTP Server อย่างปลอดภัยเพื่อป้องกันการฟิชชิ่งหรือการเปิดเผยข้อมูลระบบ 4. **สรุป** - การเปลี่ยนแปลงหลักคือการเพิ่มรายละเอียดเชิงเทคนิคเกี่ยวกับการสำรองและกู้คืนระบบ KM‑AI รวมถึงการกำหนด RTO/RPO ใหม่และการมอบหมายหน้าที่เพิ่มเติมให้กับ Softnix และ VSTECS - การแก้ไขเชิงรูปแบบ (การพิมพ์, การจัดรูปแบบ) ไม่มีความเสี่ยงที่มีนัยสำคัญ - ผู้เกี่ยวข้องควรตรวจสอบความพร้อมของกระบวนการ Backup/Restore, การตั้งค่าอีเมลแจ้งเตือน, และการยอมรับ RTO ใหม่เพื่อให้สอดคล้องกับระดับความสำคัญของบริการ### ผลกระทบจากการเปลี่ยนแปลงที่มีความหมายสำคัญ **1. มุมมองของลูกค้า (ผู้รับบริการ)** - **ตรวจสอบความพร้อมของข้อมูลสำรอง**: ลูกค้าต้องยืนยันว่า Softnix/VSTECS จะทำ Backup ตามระดับ Retention ที่ระบุ (Docker config 3 copy, Docker image 3 copy, Volume 30 วัน, LocalLLM Model 3 copy) และต้องมีแผนการเก็บข้อมูลสำรองที่สอดคล้องกับนโยบายภายในขององค์กร - **ยอมรับ RTO/RPO ใหม่**: RTO ถูกปรับจาก 4 ชม. เป็น 1 วัน ซึ่งอาจไม่สอดคล้องกับความต้องการด้านความต่อเนื่องของธุรกิจ ลูกค้าควรประเมินผลกระทบต่อ SLA ของตนและเจรจาเพื่อให้ได้ระดับ RTO ที่เหมาะสมหรือเพิ่มมาตรการสำรองเพิ่มเติม (เช่น การทำ Backup แบบต่อเนื่อง) เพื่อให้สอดคล้องกับความสำคัญของระบบ - **กำหนดกระบวนการตรวจสอบและแจ้งเตือน**: ลูกค้าต้องตรวจสอบว่าการแจ้งเตือนอีเมลจาก VSTECS (CPU, Memory, Disk, Backup/Restore) จะถูกส่งไปยังผู้รับที่เหมาะสมและมีการจัดการอีเมลอย่างปลอดภัย (การใช้ SPF/DKIM/DMARC, การเข้ารหัส TLS) เพื่อป้องกันฟิชชิ่งหรือการเปิดเผยข้อมูลระบบ - **เตรียมแผนการจัดการข้อมูลหลังการ Restore**: เนื่องจาก Volume จะทำ Full Restore ทำให้ข้อมูลที่เพิ่มหลัง Backup สุดท้ายหายไป ลูกค้าควรกำหนดกระบวนการบันทึกข้อมูลสำคัญแบบต่อเนื่องหรือทำ Backup เพิ่มเติมระหว่างช่วงเวลาเพื่อหลีกเลี่ยงการสูญเสียข้อมูลสำคัญ - **ตรวจสอบความพร้อมของฮาร์ดแวร์**: หากต้องเปลี่ยน GPU ลูกค้าต้องจัดเตรียมข้อมูลสเปคและตรวจสอบว่ามีการสนับสนุนจาก Softnix/VSTECS ในการติดตั้ง Nvidia driver อย่างถูกต้อง **2. มุมมองของผู้ให้บริการ (Softnix / VSTECS)** - **จัดทำ SOP สำหรับ Backup/Restore**: สร้างขั้นตอนปฏิบัติการที่ชัดเจนสำหรับ Internal Backup (auto‑delete/overwrite หลัง 30 วัน) และ External Backup (ตาม Destination path) พร้อมทดสอบการทำงานเป็นประจำเพื่อยืนยันความสมบูรณ์ของข้อมูลสำรอง - **กำหนดและทดสอบ RTO/RPO**: จัดทำแผนการฟื้นฟูระบบที่สามารถทำ Recovery ภายใน 1 วัน (OS 4 ชม.) และตรวจสอบให้แน่ใจว่า Recovery Point Objective 24 ชม. สามารถทำได้จริงโดยการทำ Backup อย่างน้อยทุก 24 ชม. - **จัดเตรียมระบบแจ้งเตือนอีเมล**: ตั้งค่า SMTP Server ที่ปลอดภัย, กำหนดรายการผู้รับ, กำหนด Threshold ที่เหมาะสม, และทำการทดสอบการส่งอีเมลแจ้งเตือนทุกประเภท (CPU, Memory, Disk, Backup/Restore) อย่างสม่ำเสมอ - **จัดการความปลอดภัยของข้อมูลสำรอง**: ใช้การเข้ารหัสข้อมูลสำรอง (at‑rest) และการส่งข้อมูล (in‑transit) เพื่อป้องกันการเข้าถึงโดยไม่ได้รับอนุญาต รวมถึงกำหนดสิทธิ์การเข้าถึง Backup/Restore ให้จำกัดเฉพาะบุคคลที่รับผิดชอบ - **เตรียมการสนับสนุนด้านฮาร์ดแวร์**: มีขั้นตอนและเอกสารสำหรับการจัดหาและติดตั้ง Nvidia driver เมื่อมีการเปลี่ยน GPU พร้อมการทดสอบความเข้ากันได้กับ KM‑AI System ก่อนส่งมอบให้ลูกค้า - **ทำการทดสอบ DR (Disaster Recovery) อย่างสม่ำเสมอ**: จัดทำการ Drill หรือ Simulation อย่างน้อยปีละ 1 ครั้ง เพื่อยืนยันว่า Full Restore ของ Volume สามารถทำได้โดยไม่มีข้อผิดพลาดและเวลาฟื้นฟูเป็นไปตาม RTO ที่กำหนด **สรุป**: การเปลี่ยนแปลงไม่ได้เป็นเพียงการแก้ไขรูปแบบ แต่เพิ่มข้อกำหนดเชิงเทคนิคที่สำคัญ ทั้งด้าน Backup/Restore, RTO/RPO, การแจ้งเตือน, และความรับผิดชอบของผู้ให้บริการ ลูกค้าควรตรวจสอบความสอดคล้องกับนโยบายภายในและความต้องการ SLA ส่วนผู้ให้บริการต้องจัดทำ SOP, ระบบแจ้งเตือน, ความปลอดภัยของข้อมูลสำรอง และทำการทดสอบ DR อย่างต่อเนื่องเพื่อจัดการความเสี่ยงที่เพิ่มขึ้นpage 5MODIFIED
