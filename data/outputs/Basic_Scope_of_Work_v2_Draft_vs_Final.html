สรุปภาพรวม

โดยรวมมีการเปลี่ยนแปลงจำนวน 12 รายการ (เพิ่ม 0 รายการ, ลบ 0 รายการ, แก้ไข 12 รายการ)

ส่วนที่ 1 – สรุปภาพรวมการเปลี่ยนแปลง
- หน้า 1, 2, 4, 7: แก้ไขการพิมพ์และรูปแบบหัวข้อเท่านั้น ไม่ส่งผลต่อสาระหรือความเสี่ยง (เช่น “ขอบเขตกำรให้บริกำร” → “ขอบเขตการให้บริการ”)
- หน้า 2: ลด Response Time จาก 4 ชม. เป็น 2 ชม.; เพิ่มเงื่อนไขการจัดการ Critical Incident (แจ้งผ่านหมายเลขโทรศัพท์, ระยะเวลาการแก้ไข Workaround Server Down 1 วัน, Service Down 4 ชม.); ขยายขอบเขตงานของ Data Engineer ให้ครอบคลุม Softnix Gen AI, Rocky Linux, การอัปเดตแพตช์ความปลอดภัย, Local LLMs และรายงานไตรมาส (ทำงานนอกเวลาทำการ, Remote)
- หน้า 5: เพิ่มหัวข้อ 4.2 และ 4.3 (ซ้ำ) พร้อมรายละเอียดเชิงเทคนิคของ Backup (Retention, Docker config 3 copy, Image 3 copy, Volume 30 วัน, Model 3 copy), การกู้คืน (Internal Backup เป็นหลัก, RTO ≤ 1 วัน, RPO = 24 ชม.), การจัดการ Nvidia driver, การแจ้งเตือน Monitoring (CPU/Memory/Disk Threshold) – ขยายขอบเขตความรับผิดชอบของลูกค้าและผู้ให้บริการอย่างมีนัยสำคัญ

ส่วนที่ 2 – สรุปข้อเสนอแนะภาพรวม

1) มุมมองผู้ได้รับบริการ (ลูกค้า)
- ตรวจสอบให้แน่ใจว่า SLA ใหม่ (Response 2 ชม., Workaround Server Down 1 วัน, Service Down 4 ชม.) สอดคล้องกับความต้องการและงบประมาณ; หากมีค่าใช้จ่ายเพิ่มเติมต้องเจรจาเป็นลายลักษณ์อักษร
- ขอ SOP และเอกสารการแจ้งเหตุ (Phone Number) รวมถึงรายงานไตรมาสของการอัปเดตแพตช์และความปลอดภัยจากผู้ให้บริการ
- กำหนดและทดสอบแผน DR (Recovery Test) อย่างน้อยปีละ 1 ครั้ง เพื่อยืนยันว่า Internal Backup สามารถคืนค่าได้ตาม RTO ≤ 1 วัน, RPO = 24 ชม.
- จัดทำ SOP ภายในสำหรับการตอบสนองต่อการแจ้งเตือน Monitoring และการจัดการ Nvidia driver; ตรวจสอบกระบวนการรับอีเมลแจ้งเตือนจาก VSTECS อย่างต่อเนื่อง
- ปรับสัญญา/ SLA ให้รวมข้อกำหนดเชิงเทคนิคของ Backup, Retention, Auto‑delete, และเงื่อนไขการทำงานนอกเวลาทำการ พร้อมรับรองมาตรการรักษาความปลอดภัยของการเข้าถึง Remote

2) มุมมองผู้ให้บริการ (ผู้ขาย)
- ตรวจสอบความพร้อมของทีมสนับสนุนให้ตอบสนองภายใน 2 ชม.; จัดทำระบบแจ้งเตือนและจัดสรรทรัพยากรตาม SLA ใหม่
- ปรับ SOP การจัดการ Critical Incident ให้รวมขั้นตอนแจ้งผ่านหมายเลขโทรศัพท์และระยะเวลา Workaround ตามที่ระบุในหน้า 2
- ระบุขอบเขตงานของ Data Engineer อย่างชัดเจน (Softnix Gen AI, Rocky Linux, แพตช์ความปลอดภัย, Local LLMs) พร้อมประเมินผลกระทบต่อค่าใช้จ่ายและเวลา
- จัดทำแผนการอัปเดตนอกเวลาทำการที่มีการควบคุมความปลอดภัย (Encrypted Remote Access, Log audit) และให้ลูกค้าเข้าถึงรายงานไตรมาสอย่างโปร่งใส
- พัฒนาเอกสารขั้นตอน Backup/Restore (Docker config 3 copy, Image 3 copy, Volume 30 วัน, Model 3 copy) รวมถึงสคริปต์ Auto‑delete; กำหนด SLA สำหรับการแจ้งผลสำเร็จ/ล้มเหลวของอีเมลและการตอบสนองต่อ Monitoring
- จัดทำ SOP การจัดหาและติดตั้ง Nvidia driver เมื่อมีการเปลี่ยนฮาร์ดแวร์; ทดสอบความเข้ากันได้ก่อนส่งมอบ

ส่วนที่ 3 – ผู้ที่มีส่วนได้ส่วนเสียและผลกระทบ

- ลูกค้า (ฝ่ายปฏิบัติการ IT): ต้องปรับกระบวนการตอบสนองต่อ Incident, จัดทำและทดสอบ DR, ดูแลการรับอีเมลแจ้งเตือน, ปรับงบประมาณตาม SLA ใหม่
- ลูกค้า (ฝ่ายการเงิน): ต้องประเมินค่าใช้จ่ายเพิ่มเติมจากการขยายขอบเขตงาน Data Engineer และการทำงานนอกเวลาทำการ
- ลูกค้า (ฝ่ายกฎหมาย/Compliance): ต้องตรวจสอบให้การแจ้งเหตุและการจัดการข้อมูลสอดคล้องกับ PDPA/GDPR และบันทึกเป็นลายลักษณ์อักษร
- ผู้ให้บริการ Softnix: ต้องเพิ่มทรัพยากรทีมสนับสนุน, ปรับ SOP Critical Incident, จัดทำเอกสารและสคริปต์ Backup/Restore, ควบคุมความปลอดภัยของ Remote Access
- ผู้ให้บริการ VSTECS: รับผิดชอบการจัดทำ SMTP Server, การแจ้งเตือน Monitoring, การทำ Backup/Restore ตามข้อกำหนด, และการจัดหา Nvidia driver ตามที่ลูกค้าต้องการ
- ผู้ขาย (ทีม Data Engineer): ต้องดำเนินการอัปเดต Softnix Gen AI, Rocky Linux, แพตช์ความปลอดภัย, Local LLMs ตามขอบเขตใหม่; อาจต้องขอค่าใช้จ่ายเพิ่มเติม
- ผู้จัดหาอุปกรณ์ฮาร์ดแวร์ (Nvidia, Server Vendor): มีบทบาทในการสนับสนุนการอัปเดต driver และการเปลี่ยนฮาร์ดแวร์ตาม RTO/RPO ที่กำหนด
- หน่วยงานตรวจสอบภายนอก (Auditor): อาจตรวจสอบการปฏิบัติตาม SLA, การบันทึกเหตุการณ์, และการรักษาความปลอดภัยของการเข้าถึง Remote

โดยสรุป การเปลี่ยนแปลงนี้กระทบต่อกระบวนการปฏิบัติการ, งบประมาณ, ความรับผิดชอบด้านความปลอดภัยและการปฏิบัติตามกฎระเบียบของทั้งลูกค้าและผู้ให้บริการ; จำเป็นต้องมีการทบทวนสัญญา, SOP, และแผน DR อย่างเป็นระบบเพื่อบรรเทาความเสี่ยง.

2.ขอบเขตการให้บริการ
การให้บริการภายใต้ SOW นี้ประกอบด้วย:
2.1การออกแบบและติดตั/งระบบ
2.1.1การออกแบบและกําหนดค่าระบบ
- วิเคราะห์ความต้องการของลูกค้าและออกแบบระบบ Softnix GenAI ให้เหมาะสม
-กําหนดค่าระบบตามแผนการออกแบบที่ได้รับการอนุมัติ
2.1.2 การตั้งค่า AI Agent Services
- ปรับแต่ง AI Agent Services เพื่อรองรับการใช้งานเฉพาะของลูกค้า โดยมีรายละเอียดดังนี้:
a) การปรับปรุงข้อมูล:
-เตรียมและปรับปรุงข้อมูลสําหรับAI Knowledgeจํานวนไม่เกิน10 ชุดข้อมูล
-ทําความสะอาดและจัดรูปแบบข้อมูลให้เหมาะสมสําหรับการใช้งานกับAI
b) การออกแบบ Prompt Instruction:
-สร้างและปรับแต่งคําสั่งและคําแนะนําสําหรับAI ให้เหมาะสมกับลักษณะการใช้งานของ
ลูกค้า
- ทดสอบและปรับปรุง Promptเพื่อให้ได้ผลลัพธ์ที่แม่นยําและตรงตามความต้องการ
c) การเลือกและตั้งค่า LLM:
- คัดเลือก Large Language Model (LLM) ที่เหมาะสมกับความต้องการของลูกค้า (รายการ
LLM ที่ลูกค้าใช้ได้จะขึ้นอยู่กับรุ่นของสินค้า)
- ปรับแต่งพารามิเตอร์ของ LLM เพื่อให้ได้ประสิทธิภาพสูงสุด
d) การเชื่อมต่อกับระบบภายนอก:
-ดําเนินการเชื่อมต่อกับLine Services (ลูกค้าเป็นผู้จัดหา Line Services)
- ทดสอบการเชื่อมต่อและแก้ไขปัญหาที่อาจเกิดขึ้น
e) การติดตั้ง Web Chat Application:
- ติดตั้ง Web Chat Application บนระบบของลูกค้า
- หรือ ผสานรวม Web Chat เข้ากับ Web Application ที่มีอยู่ของลูกค้า
-ทดสอบการทํางานและปรับแต่งตามความต้องการ
2.2การฝึกอบรม
2.2.1 การอบรมการใช้งานระบบ ระยะเวลา 1 วัน โดยครอบคลุมเนื้อหาดังต่อไปนี้
- ครอบคลุมหัวข้อ: การใช้งานพื้นฐาน, การจัดการผู้ใช้, การตั้งค่าระบบ, และการแก้ไขปัญหา
เบื้องต้น, หลักการของ Prompt Engineering, เทคนิคการเขียน Prompt ที่มีประสิทธิภาพ, การ
ปรับแต่ง Prompt เพื่อผลลัพธ์ที่ดีขึ้น
2.3บริการหลังการส่งมอบ
บริการหลังการส่งมอบจะมีให้ตลอดระยะเวลา Subscription โดยมีรายละเอียดดังนี้:
2.3.1การให้คําปรึกษาออนไลน์
- ให้คําปรึกษาเกี่ยวกับการใช้งานการแก้ไขปัญหา การปรับแก้ Promptและการกําหนดค่าระบบ
- บริการให้คําปรึกษาแบบ8x5 (8 ชั่วโมงต่อวัน, 5 วันต่อสัปดาห์)
- Response Time: ภายใน4ชั่วโมงหลังจากได้รับแจ้งปัญหา
- Resolution Time (Workaround) : ภายใน 24ชั่วโมงสําหรับปัญหาทั่วไป
2.3.2 บริการ Data EngineerสําหรับChange Request
• จัดสรร Data EngineerสําหรับChange Requestจํานวน5 Manday
Change Request คือคําขอให้มีการเปลี่ยนแปลงหรือปรับปรุงระบบที่ได้ติดตั้งและกําหนดค่าไว้
แล้ว โดยการเปลี่ยนแปลงนี้อยู่นอกเหนือจากขอบเขตงานเดิมที่ได้ตกลงกันไว้ในตอนเริ่มโครงการ
นอกเหนือจากบริการให้คําปรึกษาออนไลน์ได้ฟรี

Change Request คือคำขอให้มีการเปลี่ยนแปลงหรือปรับปรุงระบบที่ได้ติดตั้งและกำหนดค่าไว้
แล้ว โดยการเปลี่ยนแปลงนี้อยู่นอกเหนือจากขอบเขตงานเดิมที่ได้ตกลงกันไว้ในตอนเริ่มโครงการ
นอกเหนือจากบริการให้คำปรึกษาออนไลน์ได้ฟรี
ตัวอย่างของ Change Request ที่อาจเกิดขึ้นในโครงการ:		1. การแก้ไขหัวข้อและการพิมพ์: มีการเปลี่ยนแปลงคำสะกดและการจัดรูปแบบของหัวข้อ (เช่น "ขอบเขตกำรให้บริกำร" แทน "ขอบเขตการให้บริการ") ซึ่งเป็นการแก้ไขเชิงรูปแบบและไม่ส่งผลต่อความหมายหรือความเสี่ยงของสัญญา 2. การเปลี่ยนแปลงรายละเอียดการให้บริการหลังการส่งมอบ (2.3.1): - เวลาตอบสนอง (Response Time) ลดจาก 4 ชั่วโมงเป็น 2 ชั่วโมงหลังจากได้รับแจ้งปัญหา - เพิ่มเงื่อนไขการจัดการ Critical Incident ที่ทำให้ Server down (ไม่ใช่จาก Hardware หรือ Service Down ของผู้ใช้) พร้อมระบุวิธีการแจ้ง (หมายเลขโทรศัพท์) และระยะเวลาการแก้ไข (Workaround) สำหรับ Server down (1 วัน) และ Service Down (4 ชั่วโมง) 3. การขยายขอบเขตของบริการ Data Engineer สำหรับ Change Request (2.3.2): - เพิ่มรายการงานที่ครอบคลุมปัญหาซอฟต์แวร์แพลตฟอร์ม Softnix Gen AI, ระบบปฏิบัติการ Rocky Linux, ช่องโหว่ด้านความปลอดภัยของซอฟต์แวร์และไลบรารี, การอัปเดตแพตช์และฟีเจอร์รุ่นย่อย, การอัปเดต Local LLMs, และการส่งรายงานประจำไตรมาสเกี่ยวกับการอัปเดตแพตช์และความปลอดภัย - ระบุว่าการอัปเดตจะทำนอกเวลาทำการและสามารถทำได้แบบ Remote โดย VSTECS จะจัดเตรียม Remote ให้ - เพิ่มเงื่อนไขว่าปัญหาความปลอดภัยระดับ Critical ต้องแจ้งและอัปเดตทันที 4. การเปลี่ยนแปลงอื่น ๆ ที่เป็นการแก้ไขการพิมพ์หรือการใช้คำที่แตกต่างกัน (เช่น "กำหนดค่าระบบ" แทน "กําหนดค่าระบบ", "ทำความสะอาด" แทน "ทําความสะอาด") ไม่ได้ส่งผลต่อเนื้อหาเชิงสาระและไม่มีความเสี่ยงที่มีนัยสำคัญจากการเปลี่ยนแปลงนี้จากการเปลี่ยนแปลงที่ระบุ มีผลต่อความหมายและขอบเขตของสัญญา จึงควรดำเนินการจัดการความเสี่ยงดังนี้: 1. มุมมองของผู้ได้รับบริการ (ลูกค้า) - ตรวจสอบให้แน่ใจว่าการลดเวลา Response Time จาก 4 ชั่วโมงเป็น 2 ชั่วโมงสอดคล้องกับความต้องการของธุรกิจและมีการกำหนดค่าแรงงานหรือค่าใช้จ่ายเพิ่มเติมหรือไม่ หากมีค่าใช้จ่ายเพิ่ม ควรเจรจาต่อรองหรือขอการชดเชยที่เหมาะสม - ยืนยันขั้นตอนการแจ้งเหตุ (Phone Number) และระยะเวลาการแก้ไข (Workaround) สำหรับ Server Down (1 วัน) และ Service Down (4 ชั่วโมง) ว่าเป็นไปตามมาตรฐานการปฏิบัติงานขององค์กรและกฎหมายที่เกี่ยวข้อง (เช่น GDPR, PDPA) โดยขอเอกสาร SOP ที่ชัดเจน - ประเมินผลกระทบของการขยายขอบเขตงานของ Data Engineer (เช่น การอัปเดต Softnix Gen AI, Rocky Linux, แพตช์ความปลอดภัย, Local LLMs) ว่าจะเพิ่มภาระค่าใช้จ่ายหรือความเสี่ยงด้านความปลอดภัยหรือไม่ และขอให้ผู้ให้บริการระบุค่าใช้จ่ายเพิ่มเติมหรือเงื่อนไขการชำระเงินอย่างชัดเจน - ตรวจสอบว่าการอัปเดตนอกเวลาทำการและการทำ Remote จะมีมาตรการรักษาความปลอดภัย (เช่น การเข้ารหัส, การตรวจสอบการเข้าถึง) เพื่อลดความเสี่ยงของการละเมิดข้อมูล - กำหนดกระบวนการรับรายงานประจำไตรมาสเกี่ยวกับการอัปเดตแพตช์และความปลอดภัย เพื่อให้สามารถตรวจสอบและประเมินความเสี่ยงได้ต่อเนื่อง 2. มุมมองของผู้ให้บริการ (ผู้ขาย) - ประเมินความพร้อมของทีมสนับสนุนเพื่อให้ตอบสนองภายใน 2 ชั่วโมง รวมถึงการจัดสรรทรัพยากรและระบบแจ้งเตือนที่มีประสิทธิภาพ - ปรับปรุงขั้นตอนการจัดการ Critical Incident ให้รวมถึงการแจ้งผ่านหมายเลขโทรศัพท์ที่ระบุ และกำหนดระยะเวลาการแก้ไข (Workaround) ให้สอดคล้องกับ SLA ใหม่ พร้อมจัดทำเอกสาร SOP และฝึกอบรมทีมงาน - กำหนดขอบเขตงานของ Data Engineer อย่างชัดเจน รวมถึงรายการงานใหม่ที่เพิ่มเข้ามา (Softnix Gen AI, Rocky Linux, ความปลอดภัยระดับ Critical, การอัปเดต Local LLMs) และประเมินผลกระทบต่อค่าใช้จ่ายและเวลาในการดำเนินงาน - จัดทำแผนการอัปเดตนอกเวลาทำการที่มีการควบคุมความปลอดภัย (Remote Access, การตรวจสอบการทำงาน, การบันทึก Log) เพื่อให้ลูกค้าเชื่อมั่นในความปลอดภัยของระบบ - สร้างกระบวนการรายงานประจำไตรมาสที่เป็นมาตรฐาน รวมถึงรายละเอียดของแพตช์ที่ติดตั้ง, ช่องโหว่ที่แก้ไข, และผลกระทบต่อระบบ เพื่อให้ลูกค้าสามารถตรวจสอบและประเมินความเสี่ยงได้ -
