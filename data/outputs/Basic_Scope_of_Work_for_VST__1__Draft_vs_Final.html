สรุปภาพรวม

โดยรวมมีการเปลี่ยนแปลงจำนวน 12 รายการ (เพิ่ม 0 รายการ, ลบ 0 รายการ, แก้ไข 12 รายการ)

ส่วนที่ 1 – สรุปภาพรวมการเปลี่ยนแปลง
- แก้ไขการสะกดและรูปแบบหัวข้อ (เช่น “ขอบเขตการให้บริการ”, “การฝึกอบรม”) – ไม่ถือเป็นการเปลี่ยนแปลงสาระสำคัญ (Page 1, 2, 4, 6)
- ปรับ Response Time จาก 2 ชม. เป็น 4 ชม. และลบรายละเอียด Resolution Time สำหรับ Critical Incident, Server Down, Service Down (Page 2)
- ลบบริการด้านความปลอดภัยและการอัปเดตระบบ (Patch, Minor Version, Quarterly Report, Remote Update) รวมถึงการอ้างอิง Rocky Linux และช่องโหว่ของ Softnix Gen AI (Page 2)
- ลดขอบเขตของ Data Engineer ให้เพียงการจัดสรร 5 Manday ต่อ Change Request โดยไม่มีการรับประกันการแก้ไขช่องโหว่หรืออัปเดตซอฟต์แวร์ (Page 2)
- ลบส่วน 4.2 และ 4.3 ที่อธิบายการ Backup/Restore อย่างละเอียด ทำให้ไม่มีการระบุ SLA, RTO/RPO หรือแผน Disaster Recovery ในเอกสารใหม่ (Page 5)

ส่วนที่ 2 – ข้อเสนอแนะภาพรวม

1) มุมมองผู้ได้รับบริการ (ลูกค้า)
- ตรวจสอบและบันทึก SLA ใหม่ (Response Time 4 ชม.) พร้อมกำหนดกระบวนการภายในเพื่อเฝ้าระวังเหตุการณ์วิกฤติที่ไม่มีการรับประกัน Resolution Time
- จัดทำแผน Backup/Restore แยกเป็นเอกสารเสริม หรือพิจารณใช้ผู้ให้บริการภายนอก (เช่น VSTECS) เพื่อคุ้มครองข้อมูลและกำหนด RTO/RPO อย่างชัดเจน
- ประเมินความเสี่ยงด้านความปลอดภัยที่เกิดจากการยกเลิกบริการอัปเดตแพตช์และจัดหาเครื่องมือหรือบริการเสริมเพื่ออัปเดตระบบด้วยตนเอง
- ขอเอกสารสรุปการเปลี่ยนแปลงสัญญาเป็นลายลักษณ์อักษรเพื่อใช้เป็นหลักฐานอ้างอิงในกรณีข้อพิพาท

2) มุมมองผู้ให้บริการ (ผู้ขาย)
- ปรับสัญญาและ SLA ให้ระบุ Response Time 4 ชม. และระบุอย่างชัดเจนว่าการแก้ไข Critical Incident, Server Down, Service Down ไม่อยู่ในขอบเขตบริการ
- จัดทำแผนการสื่อสารภายในและต่อผู้ใช้บริการเพื่ออธิบายเหตุผลของการลดขอบเขตบริการและเสนอทางเลือกบริการเสริม (เช่น บริการอัปเดตแพตช์แยกค่าใช้จ่าย)
- ปรับโครงสร้างทีม Data Engineer ให้สอดคล้องกับการจัดสรร 5 Manday ต่อ Change Request และระบุขอบเขตงานอย่างชัดเจนใน SOP
- เพิ่มข้อยกเว้นความรับผิดในสัญญาเพื่อป้องกันการเรียกร้องค่าเสียหายจากการไม่มีการอัปเดตความปลอดภัยหรือ Backup/Restore

ส่วนที่ 3 – ผู้มีส่วนได้ส่วนเสียและผลกระทบ

- ลูกค้า (ฝ่ายปฏิบัติการ): ต้องรับมือกับเวลาตอบสนองที่ช้าลงและไม่มีการรับประกันการแก้ไขเหตุการณ์วิกฤติ; ต้องจัดทำหรือจ้างบริการ Backup/Restore เอง
- ลูกค้า (ฝ่ายความปลอดภัย/Compliance): สูญเสียการรับประกันการอัปเดตแพตช์และการรายงาน Quarterly ทำให้ความเสี่ยงด้านช่องโหว่เพิ่มขึ้น
- ทีม Data Engineer ของผู้ขาย: งานถูกจำกัดให้เพียง 5 Manday ต่อ Change Request ลดภาระงานแต่ต้องชี้แจงขอบเขตที่ไม่รวมการแก้ไขช่องโหว่หรืออัปเดตซอฟต์แวร์
- ผู้ให้บริการ Backup ภายนอก (เช่น VSTECS): มีโอกาสได้รับการว่าจ้างเพิ่มเติมจากลูกค้าที่ต้องการบริการสำรองข้อมูลที่ถูกลบออกจากสัญญาเดิม
- ฝ่ายบริหารของผู้ขาย: ต้องประเมินผลกระทบต่อชื่อเสียงและความพึงพอใจของลูกค้า เนื่องจากการลดขอบเขตบริการอาจทำให้ลูกค้าเปลี่ยนผู้ให้บริการในอนาคต
- ฝ่ายกฎหมาย/สัญญาของทั้งสองฝ่าย: ต้องทบทวนและปรับปรุงข้อกำหนดสัญญาเพื่อให้สอดคล้องกับการเปลี่ยนแปลงและลดความเสี่ยงจากข้อพิพาทในอนาคต

2.ขอบเขตกำรให้บริกำร
การให้บริการภายใต้ SOW นี้ประกอบด้วย:
2.1กำรออกแบบและติดตั้งระบบ
2.1.1การออกแบบและกำหนดค่าระบบ
- วิเคราะห์ความต้องการของลูกค้าและออกแบบระบบ Softnix GenAI ให้เหมาะสม
-กำหนดค่าระบบตามแผนการออกแบบที่ได้รับการอนุมัติ
2.1.2 การตั้งค่า AI Agent Services
- ปรับแต่ง AI Agent Services เพื่อรองรับการใช้งานเฉพาะของลูกค้า โดยมีรายละเอียดดังนี้:
a) การปรับปรุงข้อมูล:
-เตรียมและปรับปรุงข้อมูลสำหรับAI Knowledgeจำนวนไม่เกิน10 ชุดข้อมูล
-ทำความสะอาดและจัดรูปแบบข้อมูลให้เหมาะสมสำหรับการใช้งานกับAI
b) การออกแบบ Prompt Instruction:
-สร้างและปรับแต่งคำสั่งและคำแนะนำสำหรับAI ให้เหมาะสมกับลักษณะการใช้งานของ
ลูกค้า
- ทดสอบและปรับปรุง Promptเพื่อให้ได้ผลลัพธ์ที่แม่นยำและตรงตามความต้องการ
c) การเลือกและตั้งค่า LLM:
- คัดเลือก Large Language Model (LLM) ที่เหมาะสมกับความต้องการของลูกค้า (รายการ
LLM ที่ลูกค้าใช้ได้จะขึ้นอยู่กับรุ่นของสินค้า)
- ปรับแต่งพารามิเตอร์ของ LLM เพื่อให้ได้ประสิทธิภาพสูงสุด
d) การเชื่อมต่อกับระบบภายนอก:
-ดำเนินการเชื่อมต่อกับLine Services (ลูกค้าเป็นผู้จัดหา Line Services)
- ทดสอบการเชื่อมต่อและแก้ไขปัญหาที่อาจเกิดขึ้น
e) การติดตั้ง Web Chat Application:
- ติดตั้ง Web Chat Application บนระบบของลูกค้า
- หรือ ผสานรวม Web Chat เข้ากับ Web Application ที่มีอยู่ของลูกค้า
-ทดสอบการทำงานและปรับแต่งตามความต้องการ
2.2กำรฝึกอบรม
2.2.1 การอบรมการใช้งานระบบ ระยะเวลา 1 วัน โดยครอบคลุมเนื้อหาดังต่อไปนี้
- ครอบคลุมหัวข้อ: การใช้งานพื้นฐาน, การจัดการผู้ใช้, การตั้งค่าระบบ, และการแก้ไขปัญหา
เบื้องต้น, หลักการของ Prompt Engineering, เทคนิคการเขียน Prompt ที่มีประสิทธิภาพ, การ
ปรับแต่ง Prompt เพื่อผลลัพธ์ที่ดีขึ้น
2.3บริกำรหลังกำรส่งมอบ

- Server down ที่ไม่ใช่ปัญหาจาก Hardware : 1 วัน

- ทดสอบและปรับปรุง Promptเพื่อให้ได้ผลลัพธ์ที่แม่นยําและตรงตามความต้องการ
c) การเลือกและตั้งค่า LLM:

LLM ที่ลูกค้าใช้ได้จะขึ้นอยู่กับรุ่นของสินค้า)
- ปรับแต่งพารามิเตอร์ของ LLM เพื่อให้ได้ประสิทธิภาพสูงสุด

2.3.2 บริการ Data EngineerสําหรับChange Request
• จัดสรร Data Engineerสําหรับ Change Request จํานวน5 Manday
Change Request คือคําขอให้มีการเปลี่ยนแปลงหรือปรับปรุงระบบที่ได้ติดตั้งและกําหนดค่าไว้
แล้ว โดยการเปลี่ยนแปลงนี้อยู่นอกเหนือจากขอบเขตงานเดิมที่ได้ตกลงกันไว้ในตอนเริ่มโครงการ
นอกเหนือจากบริการให้คําปรึกษาออนไลน์ได้ฟรี
ตัวอย่างของ Change Request ที่อาจเกิดขึ้นในโครงการ:		### การเปลี่ยนแปลงที่สำคัญระหว่างข้อความเดิมและข้อความใหม่ 1. **หัวข้อและการสะกด** – มีการแก้ไขคำบางคำเพื่อให้เป็นภาษาไทยที่ถูกต้อง (เช่น "ขอบเขตการให้บริการ" แทน "ขอบเขตกำรให้บริกำร") แต่ตามเงื่อนไขข้อ 4 ไม่ต้องอธิบายรายละเอียดของการแก้ไขการพิมพ์หรือการสะกดคำเหล่านี้ 2. **การกำหนดขอบเขตบริการหลังการส่งมอบ** - **Response Time**: จากเดิม "ภายใน 2 ชั่วโมง" ลดลงเป็น "ภายใน 4 ชั่วโมง" - **Resolution Time (Workaround)**: ยังคงอยู่ที่ 24 ชั่วโมงสำหรับปัญหาทั่วไป แต่ส่วนที่อธิบายเวลาการแก้ไขสำหรับ **Critical Incident**, **Server down**, **Service Down** ถูกลบออกทั้งหมด - **รายละเอียดการให้บริการด้านความปลอดภัยและการอัปเดต** ถูกตัดออกอย่างสมบูรณ์ ได้แก่ * การให้บริการ Update Patch, New Feature สำหรับ Minor Version ของ Softnix Gen AI และ OS Rocky Linux * การอัปเดต Version ของ Local LLMs * การส่ง Report ทุก Quarter เกี่ยวกับ Patch และ Security Update * การอัปเดตนอกเวลาทำการ (Remote update) และการจัดเตรียม Remote ให้ VSTECS - **การอ้างอิงระบบปฏิบัติการ** (Rocky Linux) และการอธิบายช่องโหว่ด้านความปลอดภัยของซอฟต์แวร์แพลตฟอร์ม Softnix Gen AI ถูกลบออก ทำให้ขอบเขตการรับผิดชอบของ Data Engineer ลดลงอย่างชัดเจน 3. **บริการ Data Engineer สำหรับ Change Request** - ข้อความเดิมระบุรายละเอียดครบถ้วนของขอบเขตงาน (การแก้ไขปัญหาซอฟต์แวร์, ช่องโหว่ความปลอดภัย, การอัปเดตแพตช์, รายงาน Quarterly, การอัปเดตนอกเวลาทำการ ฯลฯ) - ข้อความใหม่เหลือเพียงการจัดสรร Data Engineer จำนวน 5 Manday และคำนิยามของ Change Request เท่านั้น; รายละเอียดเชิงเทคนิคและบริการเพิ่มเติมทั้งหมดถูกลบออก 4. **การจัดรูปแบบหัวข้อย่อย** – มีการเปลี่ยนแปลงบางส่วนของการจัดวางข้อความ (เช่น "การออกแบบและติดตั/งระบบ" แทน "การออกแบบและติดตั้งระบบ") ซึ่งเป็นการแก้ไขเชิงรูปแบบและไม่ถือเป็นการเปลี่ยนแปลงสาระสำคัญ 5. **ส่วนของการฝึกอบรม** – ไม่มีการเปลี่ยนแปลงสาระสำคัญ; เพียงแต่เปลี่ยนชื่อหัวข้อจาก "กำรฝึกอบรม" เป็น "การฝึกอบรม" ซึ่งเป็นการแก้ไขเชิงรูปแบบ ### สรุปผลกระทบ - **ขอบเขตการให้บริการหลังการส่งมอบ** ถูกลดลงอย่างมีนัยสำคัญ: เวลาตอบสนองช้าลง, การจัดการเหตุการณ์วิกฤติและการอัปเดตระบบ/ความปลอดภัยถูกตัดออก, ไม่มีการรายงาน Quarterly หรือการอัปเดตนอกเวลาทำการ - **บทบาทของ Data Engineer** ถูกจำกัดให้เพียงการจัดสรรเวลา (5 Manday) โดยไม่มีการรับประกันการแก้ไขช่องโหว่หรือการอัปเดตซอฟต์แวร์/OS ใด ๆ - ลูกค้าจะได้รับการสนับสนุนที่น้อยลงในด้านความปลอดภัยและการบำรุงรักษาระบบเมื่อเทียบกับเวอร์ชันเดิม - การเปลี่ยนแปลงเหล่านี้อาจส่งผลต่อความเสี่ยงด้านการดำเนินงานและความปลอดภัยของระบบที่ลูกค้าใช้งาน เนื่องจากไม่มีการรับประกันการอัปเดตแพตช์หรือการจัดการเหตุการณ์วิกฤติที่เคยระบุไว้ในสัญญาเดิม1. มุมมองลูกค้า (ผู้รับบริการ) - ตรวจสอบผลกระทบต่อความเสี่ยงด้านความปลอดภัยและการดำเนินงานของระบบที่อาจเกิดจากการยกเลิกบริการอัปเดตแพตช์และการจัดการเหตุการณ์วิกฤติ; หากความเสี่ยงสูง ควรพิจารณาต่อรองเพื่อเพิ่มระดับการสนับสนุนหรือจัดหาแหล่งบริการเสริมอื่น - ปรับแผนการจัดการความต่อเนื่องของธุรกิจ (BCP) ให้สอดคล้องกับ SLA ใหม่ (Response Time 4 ชม.) และไม่มีการรับประกัน Resolution Time สำหรับเหตุการณ์วิกฤติ; กำหนดขอบเขตงานที่ชัดเจนและระบุว่าการแก้ไขช่องโหว่หรืออัปเดตซอฟต์แวร์ไม่อยู่ในขอบเขตนี้ - ประเมินความเสี่ยงด้านความรับผิดชอบและพิจารณการเพิ่มข้อยกเว้นหรือจำกัดความรับผิดในสัญญา เพื่อป้องกันการเรียกร้องค่าเสียหายจากการไม่มีการอัปเดตความปลอดภัย - จัดทำกระบวนการตรวจสอบภายในเพื่อให้แน่ใจว่าการให้บริการตาม SLA ใหม่เป็นไปตามมาตรฐานคุณภาพและไม่ส่งผลเสียต่อชื่อเสียงของบริษัทpage 2MODIFIED

-จัดเตรียมข้อมูลที่จำเป็นสำหรับการสร้างAI Knowledge
- จัดหาและให้สิทธิ์การเข้าถึงระบบที่เกี่ยวข้อง เช่น Line Services
-จัดสรรบุคลากรที่เกี่ยวข้องเพื่อเข้าร่วมการฝึกอบรมและให้ข้อมูลที่จำเป็น
- แจ้งปัญหาหรือข้อขัดข้องให้ทีมบริการทราบโดยเร็วที่สุด
4.2 Backup KM-AI System
• การ Backup แบบ Retention ข้อมูลดังนี้
• Docker config 3 Copy
• Docker Image 3 Copy
• Docker Volume 30 Day
• LocalLLM Model 3 Copy
• การ Backup แบบ Internal Backup
• อ้างอิง Retention จากข้อด้านบน
• Script auto delete หรือ เขียนทับเมื่อครบ กำหนด retention 30 วัน
• Alert email กรณี Back up Success หรือ Fail
• External backup ทาง VSTECS จะเป็นผู้ดูแลโดย จะไป backup ตาม Destination
path ที่ Softnix แจ้งไว้ในเอกสาร Config summary system KM-AI VST
4.3 Restore KM-AI System
• จะเป็นการ Restore จาก Internal Backup
• การใช้ External backup ใช้เมื่อ ไม่สามารถใช้งานข้อมูลจาก Internal Backup ได้
• การ Restore ในส่วนของ Data (Volume) จะเป็น แบบ Full Restore เป็นผลให้ข้อมูล
ที่เพิ่มเข้าไปหลังการ backup จะหายทั้งหมด
• การ Restore ระดับ compose, image, LLM จะไม่กระทบกับ Data Volume
• Nvidia driver ในกรณีที่มีการเปลี่ยนการ์ด เนื่องจาก Hardware
• เสีย Softnix จะเป็นผู้ระบุ รุ่นการ์ด ที่ Support ที่ทำให้ OS มองเห็นหรือใช้งานได้
โดย VSTECS จะเป็นผู้จัดหา และทาง Softnix จะเป็นผู้ดำเนินการติดตั้ง Nvidia
driver จนสามรถกลับมาใช้งานได้
• Recovery Time(RTO) ในกรณีเครื่อง เสียหาย จำเป็นต้อง Install และ โดยเริ่ม
หลังจากที่ HPE เปลี่ยน Hardware เรียบร้อยภายใน 1 วัน
• Recovery point (RPO) เท่ากับ 24 ชั่วโมง

เพื่อให้การดําเนินโครงการเป็นไปอย่างราบรื่นลูกค้าจะต้องรับผิดชอบในเรื่องต่อไปนี้:
-จัดเตรียมข้อมูลที่จําเป็นสําหรับการสร้างAI Knowledge
- จัดหาและให้สิทธิ์การเข้าถึงระบบที่เกี่ยวข้อง เช่น Line Services
-จัดสรรบุคลากรที่เกี่ยวข้องเพื่อเข้าร่วมการฝึกอบรมและให้ข้อมูลที่จําเป็น
- แจ้งปัญหาหรือข้อขัดข้องให้ทีมบริการทราบโดยเร็วที่สุด		การเปลี่ยนแปลงหลักระหว่างข้อความเดิมและข้อความใหม่คือ: 1. **การลบเนื้อหาส่วนสำคัญเกี่ยวกับการสำรองข้อมูล (Backup) และการกู้คืนข้อมูล (Restore)** – ส่วน 4.2 และ 4.3 ของข้อความเดิมที่อธิบายรายละเอียดเชิงเทคนิคของการ Backup (Retention, จำนวนสำเนา, ระยะเวลาการเก็บ, การแจ้งเตือนอีเมล, การใช้ External Backup ของ VSTECS) และการ Restore (วิธีการ Restore จาก Internal Backup, เงื่อนไขการใช้ External Backup, ผลของ Full Restore ต่อ Data Volume, การจัดการ Nvidia driver, RTO/RPO, การมอนิเตอร์ระบบ) ถูกตัดออกทั้งหมดในข้อความใหม่ ทำให้ความรับผิดชอบของลูกค้าในด้านการจัดการข้อมูลสำรองและการกู้คืนระบบไม่ปรากฏอยู่แล้ว 2. **การปรับหัวข้อ** – จาก "4. ควำมรับผิดชอบของลูกค้ำ" เป็น "4. ความรับผิดชอบของลูกค้า" (เป็นการแก้ไขการสะกดและรูปแบบเพียงเล็กน้อย) ซึ่งเป็นการเปลี่ยนแปลงเชิงรูปแบบที่ไม่มีผลต่อสาระสำคัญ 3. **การปรับข้อความในรายการข้อรับผิดชอบ** – รายการ 4.1 ของข้อความเดิมยังคงอยู่ในข้อความใหม่โดยมีการแก้ไขการพิมพ์และการใช้คำบางส่วน (เช่น "ข้อมูลที่จำเป็นสำหรับการสร้าง AI Knowledge" เป็น "ข้อมูลที่จําเป็นสําหรับการสร้าง AI Knowledge") ซึ่งเป็นการแก้ไขเชิงรูปแบบและไม่มีผลต่อความหมาย **ผลกระทบและความเสี่ยงที่อาจเกิดขึ้น** - การลบส่วนที่เกี่ยวกับ Backup/Restore ทำให้ขอบเขตความรับผิดชอบของลูกค้าในด้านการรักษาความต่อเนื่องของระบบและการปกป้องข้อมูลไม่ชัดเจน อีกทั้งอาจทำให้ข้อกำหนด SLA, RTO/RPO, และแผนฟื้นฟูระบบ (Disaster Recovery) ไม่ถูกบันทึกไว้ในสัญญา ส่งผลให้ความเสี่ยงต่อการสูญเสียข้อมูลหรือการหยุดทำงานของระบบเพิ่มขึ้น - ความรับผิดชอบต่อการจัดหาและดูแล Nvidia driver, การเปลี่ยนฮาร์ดแวร์, และการมอนิเตอร์ระบบ (CPU/Memory/Disk Threshold, Backup/Restore success alerts) ที่เคยระบุไว้ในข้อความเดิม ไม่ได้ถูกกล่าวถึงในข้อความใหม่ ทำให้ผู้รับผิดชอบ (ลูกค้า หรือผู้ให้บริการ) อาจเกิดความสับสนหรือขาดการประสานงานในกรณีที่เกิดปัญหาเทคนิค - หากไม่มีการระบุ RTO/RPO อย่างชัดเจนในเอกสารใหม่ ลูกค้าอาจไม่มีข้อมูลในการประเมินความพร้อมของระบบต่อเหตุฉุกเฉิน ซึ่งอาจส่งผลต่อการวางแผนธุรกิจและการจัดสรรทรัพยากร **สรุป**: การเปลี่ยนแปลงที่สำคัญคือการลบเนื้อหาที่เกี่ยวกับการสำรองและกู้คืนข้อมูล รวมถึงรายละเอียดเชิงเทคนิคและมาตรการการมอนิเตอร์ ซึ่งเป็นการลดความชัดเจนของขอบเขตความรับผิดชอบของลูกค้าในด้านการจัดการข้อมูลและความต่อเนื่องของระบบ ส่วนการแก้ไขการพิมพ์และหัวข้อเป็นการเปลี่ยนแปลงเชิงรูปแบบที่ไม่มีความเสี่ยงสำคัญการลบส่วน 4.2 และ 4.3 ที่อธิบายรายละเอียดการสำรองและกู้คืนข้อมูลเป็นการเปลี่ยนแปลงที่มีผลต่อความหมายและขอบเขตความรับผิดชอบของลูกค้าอย่างสำคัญ ดังนั้นจึงต้องดำเนินการจัดการความเสี่ยงต่อไปนี้: **มุมมองลูกค้า** 1. ตรวจสอบและบันทึกข้อตกลงด้าน SLA, RTO/RPO, และแผน Disaster Recovery แยกเป็นเอกสารเสริม หรือขอให้ผู้ให้บริการเพิ่มข้อกำหนดเหล่านี้ในสัญญาใหม่เพื่อให้แน่ใจว่ามีการรับรองการสำรองและกู้คืนข้อมูลอย่างชัดเจน 2. จัดทำนโยบายการสำรองข้อมูลภายในองค์กร (Retention, จำนวนสำเนา, ระยะเวลาการเก็บ, การแจ้งเตือน) และกำหนดผู้รับผิดชอบด้านเทคนิค (เช่น การจัดการ Nvidia driver, การตรวจสอบ Disk/CPU thresholds) 3. ตั้งระบบมอนิเตอร์และแจ้งเตือนอิสระ (เช่น ระบบอีเมลหรือ SMS) เพื่อรับรู้สถานะการสำรอง/กู้คืน หากผู้ให้บริการไม่ให้บริการนี้ 4. ประเมินความเสี่ยงต่อการสูญเสียข้อมูลและกำหนดแผนสำรองฉุกเฉิน (เช่น การสำรองข้อมูลสำคัญไปยังคลาวด์หรืออุปกรณ์ภายนอก) เพื่อป้องกันการหยุดทำงานของระบบ **มุมมองผู้ให้บริการ** 1. ทบทวนสัญญาและเพิ่มข้อกำหนดเกี่ยวกับ Backup/Restore (Retention, จำนวนสำเนา, ระยะเวลาเก็บ, การแจ้งเตือน) ให้ชัดเจน รวมถึงการกำหนด RTO/RPO ที่สอดคล้องกับความต้องการของลูกค้า 2. จัดทำเอกสาร SOP (Standard Operating Procedure) สำหรับการสำรองและกู้คืนข้อมูล รวมถึงขั้นตอนการจัดการ Nvidia driver, การเปลี่ยนฮาร์ดแวร์, และการมอนิเตอร์ระบบ 3. ให้บริการหรือแนะนำเครื่องมือมอนิเตอร์อิสระ (เช่น Grafana, Prometheus) เพื่อให้ลูกค้าสามารถตรวจสอบสถานะการสำรองและประสิทธิภาพระบบได้เอง 4. กำหนดกระบวนการแจ้งเตือนและรายงานผลการสำรอง/กู้คืนเป็นประจำ (รายวัน/สัปดาห์) เพื่อแสดงความรับผิดชอบและลดความไม่แน่นอนของลูกค้า 5. พิจารณาเพิ่มบริการ External Backup (เช่น VSTECS) เป็นตัวเลือกเสริมในสัญญา หรือให้คำปรึกษาเรื่องการเลือกผู้ให้บริการสำรองข้อมูลภายนอก โดยสรุป ทั้งสองฝ่ายควรทำการบันทึกและสื่อสารขอบเขตความรับผิดชอบด้าน Backup/Restore อย่างเป็นลายลักษณ์อักษร เพื่อป้องกันความสับสนและลดความเสี่ยงต่อการสูญเสียข้อมูลหรือการหยุดทำงานของระบบpage 5MODIFIED
