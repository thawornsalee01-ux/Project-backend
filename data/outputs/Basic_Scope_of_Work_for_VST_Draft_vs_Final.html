ส่วนที่ 1: สรุปภาพรวมการเปลี่ยนแปลง
- Page 2: เวลา Response Time ของบริการให้คำปรึกษาออนไลน์เปลี่ยนจาก “ภายใน 2 ชม.” เป็น “ภายใน 4 ชม.” (เพิ่ม 2 ชม.)
- Page 2: ข้อกำหนด SLA ของ Critical Incident (Server down 1 วัน, Service Down 4 ชม.) และหมายเลขโทรศัพท์แจ้งเหตุฉุกเฉินถูกลบทั้งหมด ทำให้ขอบเขตการรับผิดชอบไม่ชัดเจน
- Page 2: ขอบเขตบริการ Data Engineer ลดจากหลายรายการเป็น “จัดสรร Data Engineer สำหรับ Change Request จำนวน 5 Manday” เท่านั้น
- Page 5: ส่วนการสำรองและกู้คืนระบบ KM‑AI (Backup/Restore, RTO/RPO, การแจ้งเตือน) ถูกตัดออกทั้งหมด ทำให้ไม่มีข้อกำหนดด้านการกู้คืนข้อมูลในสัญญาใหม่

ส่วนที่ 2: สรุปข้อเสนอแนะภาพรวม

1) มุมมองผู้ได้รับบริการ (ลูกค้า)
- ตรวจสอบว่าการเพิ่ม Response Time เป็น 4 ชม. ตรงกับความต้องการของธุรกิจหรือไม่; หากไม่ควรต่อรองให้คืน SLA เดิมหรือขอ Service Credit
- ขอให้ผู้ให้บริการจัดทำ Addendum ระบุขั้นตอนและช่องทางการแจ้งเหตุฉุกเฉิน (รวมหมายเลขโทรศัพท์หรือระบบ Ticket) อย่างชัดเจน
- ตรวจสอบว่าขอบเขต Data Engineer 5 Manday ครอบคลุมความต้องการด้าน Patch Management, Security Update, LLM Update หรือไม่; หากไม่ควรขอเพิ่มบริการหรือทำสัญญาเสริม
- ดำเนินการเพิ่มข้อกำหนด Backup/Restore (RPO ≤ 24 ชม., RTO ≤ 1 วัน) พร้อมแผน Disaster Recovery และการทดสอบประจำปี

2) มุมมองผู้ให้บริการ (ผู้ขาย)
- จัดทำ Addendum อย่างเป็นทางการเพื่อบันทึกการเปลี่ยนแปลง SLA (Response Time 4 ชม.) และอธิบายเหตุผลให้สอดคล้องกับกฎหมายและ ISO/IEC 27001
- กำหนดกระบวนการแจ้งเหตุฉุกเฉินใหม่ (ช่องทางติดต่อ, เวลาตอบสนอง, ระยะเวลาการแก้ไข) และสื่อสารให้ลูกค้าทราบ; พิจารณาเพิ่มช่องทางดิจิทัล (e‑mail, portal) หากไม่มีหมายเลขโทรศัพท์
- เสนอแพคเกจเสริมสำหรับ Data Engineer (Patch Management, Security Update) พร้อมระบุขอบเขตและค่าใช้จ่ายอย่างโปร่งใส
- หากลูกค้าต้องการบริการ Backup/Restore ให้จัดทำ Annex หรือ Service Add‑on ที่ระบุเทคนิค, SLA, ค่าใช้จ่ายและความรับผิดชอบ; พิจารณาเพิ่ม Limitation of Liability ที่สอดคล้องกับมาตรฐานอุตสาหกรรม

ส่วนที่ 3: ผู้มีส่วนได้ส่วนเสียและผลกระทบ
- ลูกค้า: อาจประสบความล่าช้าในการรับบริการ (Response Time เพิ่ม), ความเสี่ยงต่อการหยุดทำงานและช่องโหว่ความปลอดภัยเพิ่มขึ้น, ขาดความคุ้มครองข้อมูลจากการลบ Backup/Restore, ต้องเจรจาเพิ่มค่าใช้จ่ายหรือทำ Addendum เพื่อรักษาระดับบริการเดิม
- ผู้ให้บริการ: ลดภาระงานและต้นทุน (Response Time ยาวขึ้น, ขอบเขต Data Engineer ลด) แต่ต้องรับความเสี่ยงด้านการละเมิด SLA, ความเสี่ยงทางกฎหมายจากการไม่มีข้อกำหนด Critical Incident และ Backup/Restore, จำเป็นต้องสื่อสารและจัดทำเอกสารเสริมเพื่อป้องกันข้อพิพาทในอนาคต
- ทีมสนับสนุนและการจัดการเหตุฉุกเฉิน: ต้องปรับกระบวนการทำงานตาม SLA ใหม่และอาจต้องพัฒนาช่องทางติดต่อใหม่
- ผู้ตรวจสอบ (Audit/Compliance): ต้องตรวจสอบว่าการเปลี่ยนแปลงสอดคล้องกับมาตรฐาน ISO/IEC 27001 และข้อกำหนดกฎหมายคุ้มครองผู้บริโภค.

2.ขอบเขตกำรให้บริกำร
การให้บริการภายใต้ SOW นี้ประกอบด้วย:
2.1กำรออกแบบและติดตั้งระบบ
2.1.1การออกแบบและกำหนดค่าระบบ
- วิเคราะห์ความต้องการของลูกค้าและออกแบบระบบ Softnix GenAI ให้เหมาะสม
-กำหนดค่าระบบตามแผนการออกแบบที่ได้รับการอนุมัติ
2.1.2 การตั้งค่า AI Agent Services
- ปรับแต่ง AI Agent Services เพื่อรองรับการใช้งานเฉพาะของลูกค้า โดยมีรายละเอียดดังนี้:
a) การปรับปรุงข้อมูล:
-เตรียมและปรับปรุงข้อมูลสำหรับAI Knowledgeจำนวนไม่เกิน10 ชุดข้อมูล
-ทำความสะอาดและจัดรูปแบบข้อมูลให้เหมาะสมสำหรับการใช้งานกับAI
b) การออกแบบ Prompt Instruction:
-สร้างและปรับแต่งคำสั่งและคำแนะนำสำหรับAI ให้เหมาะสมกับลักษณะการใช้งานของ
ลูกค้า
- ทดสอบและปรับปรุง Promptเพื่อให้ได้ผลลัพธ์ที่แม่นยำและตรงตามความต้องการ
c) การเลือกและตั้งค่า LLM:
- คัดเลือก Large Language Model (LLM) ที่เหมาะสมกับความต้องการของลูกค้า (รายการ
LLM ที่ลูกค้าใช้ได้จะขึ้นอยู่กับรุ่นของสินค้า)
- ปรับแต่งพารามิเตอร์ของ LLM เพื่อให้ได้ประสิทธิภาพสูงสุด
d) การเชื่อมต่อกับระบบภายนอก:
-ดำเนินการเชื่อมต่อกับLine Services (ลูกค้าเป็นผู้จัดหา Line Services)
- ทดสอบการเชื่อมต่อและแก้ไขปัญหาที่อาจเกิดขึ้น
e) การติดตั้ง Web Chat Application:
- ติดตั้ง Web Chat Application บนระบบของลูกค้า
- หรือ ผสานรวม Web Chat เข้ากับ Web Application ที่มีอยู่ของลูกค้า
-ทดสอบการทำงานและปรับแต่งตามความต้องการ
2.2กำรฝึกอบรม
2.2.1 การอบรมการใช้งานระบบ ระยะเวลา 1 วัน โดยครอบคลุมเนื้อหาดังต่อไปนี้
- ครอบคลุมหัวข้อ: การใช้งานพื้นฐาน, การจัดการผู้ใช้, การตั้งค่าระบบ, และการแก้ไขปัญหา
เบื้องต้น, หลักการของ Prompt Engineering, เทคนิคการเขียน Prompt ที่มีประสิทธิภาพ, การ
ปรับแต่ง Prompt เพื่อผลลัพธ์ที่ดีขึ้น

2.3บริการหลังการส่งมอบ
บริการหลังการส่งมอบจะมีให้ตลอดระยะเวลา Subscription โดยมีรายละเอียดดังนี้:
2.3.1การให้คําปรึกษาออนไลน์
- ให้คําปรึกษาเกี่ยวกับการใช้งานการแก้ไขปัญหา การปรับแก้ Promptและการกําหนดค่าระบบ
- บริการให้คําปรึกษาแบบ8x5 (8 ชั่วโมงต่อวัน, 5 วันต่อสัปดาห์)
- Response Time: ภายใน4ชั่วโมงหลังจากได้รับแจ้งปัญหา
- Resolution Time (Workaround) : ภายใน 24ชั่วโมงสําหรับปัญหาทั่วไป
2.3.2 บริการ Data EngineerสําหรับChange Request
• จัดสรร Data Engineerสําหรับ Change Request จํานวน5 Manday
Change Request คือคําขอให้มีการเปลี่ยนแปลงหรือปรับปรุงระบบที่ได้ติดตั้งและกําหนดค่าไว้
แล้ว โดยการเปลี่ยนแปลงนี้อยู่นอกเหนือจากขอบเขตงานเดิมที่ได้ตกลงกันไว้ในตอนเริ่มโครงการ
นอกเหนือจากบริการให้คําปรึกษาออนไลน์ได้ฟรี
ตัวอย่างของ Change Request ที่อาจเกิดขึ้นในโครงการ:		### การเปลี่ยนแปลงที่สำคัญระหว่างข้อความเดิมและข้อความใหม่ 1. **การแก้ไขเชิงรูปแบบและการพิมพ์** - มีการแก้ไขคำสะกดและการจัดรูปแบบหลายจุด (เช่น "ขอบเขตกำรให้บริกำร" → "ขอบเขตการให้บริการ", "การออกแบบและติดตั/งระบบ") ซึ่งเป็นการแก้ไขเชิงรูปแบบเท่านั้น **ไม่พบความเสี่ยงที่มีนัยสำคัญจากการเปลี่ยนแปลงนี้** 2. **การเปลี่ยนแปลงเนื้อหาสาระสำคัญ** - **Response Time** ของบริการให้คำปรึกษาออนไลน์เปลี่ยนจาก **ภายใน 2 ชั่วโมง** เป็น **ภายใน 4 ชั่วโมง** หลังจากได้รับแจ้งปัญหา ซึ่งทำให้ระดับการให้บริการลดลง 2 ชั่วโมงและอาจส่งผลต่อความพึงพอใจของลูกค้าและความเสี่ยงต่อการเกิดความล่าช้าในการแก้ไขปัญหา - **ส่วนของการจัดการเหตุการณ์วิกฤต (Critical Incident)** ที่เคยระบุรายละเอียดเช่น "Server down ที่ไม่ใช่ปัญหาจาก Hardware : 1 วัน" และ "Service Down ใช้เวลา 4 ชั่วโมง" รวมถึงหมายเลขโทรศัพท์สำหรับแจ้งเหตุการณ์ ถูก **ลบออกทั้งหมด** จากข้อความใหม่ ทำให้ขอบเขตการรับผิดชอบและ SLA ของเหตุการณ์วิกฤตไม่ชัดเจนอีกต่อไป - **รายละเอียดของบริการ Data Engineer สำหรับ Change Request** ถูกลดทอนอย่างมาก: - ข้อความเดิมระบุรายการบริการหลายรายการ ได้แก่ การแก้ไขปัญหาซอฟท์แวร์, การอัปเดตแพตช์ความปลอดภัยของแพลตฟอร์ม Softnix Gen AI และ OS Rocky Linux, การอัปเดต LLMs, การส่งรายงาน Quarterly, การอัปเดตนอกเวลาทำการ ฯลฯ - ข้อความใหม่เหลือเพียง "จัดสรร Data Engineer สำหรับ Change Request จำนวน 5 Manday" พร้อมคำอธิบายสั้น ๆ ของ Change Request เท่านั้น - การตัดทอนนี้ทำให้ขอบเขตการให้บริการของ Data Engineer ลดลงอย่างชัดเจนและอาจทำให้ลูกค้าไม่ได้รับการสนับสนุนด้านความปลอดภัยและการอัปเดตซอฟท์แวร์ตามที่เคยสัญญาไว้ - **การระบุช่องทางการติดต่อ (หมายเลขโทรศัพท์)** ถูกลบออกจากส่วนของบริการให้คำปรึกษาออนไลน์ ทำให้ช่องทางการแจ้งปัญหาแบบเรียกเข้าไม่มีการระบุชัดเจนในเอกสาร 3. **ผลกระทบทางกฎหมายและการบริหารโครงการ** - การเปลี่ยนแปลง SLA (Response Time จาก 2h → 4h) และการลบข้อกำหนดเกี่ยวกับ Critical Incident ทำให้สัญญาเดิมอาจต้องมีการแก้ไขเพิ่มเติม (Addendum) เพื่อให้แน่ใจว่าข้อกำหนดใหม่สอดคล้องกับความคาดหวังของลูกค้าและไม่ขัดกับข้อผูกพันเดิมของผู้ให้บริการ - การลดขอบเขตของบริการ Data Engineer อาจทำให้ข้อกำหนดในส่วนของ "Change Request" ของสัญญาเดิมไม่ครบถ้วน หากลูกค้าต้องการบริการที่เคยระบุไว้ในข้อความเดิม พวกเขาอาจต้องเจรจาเพิ่มเติมหรือทำสัญญาเสริมเพื่อคุ้มครองความเสี่ยงด้านความปลอดภัยและการอัปเดตซอฟท์แวร์ - การลบหมายเลขโทรศัพท์และรายละเอียดการแจ้งเหตุการณ์วิกฤตอาจทำให้การจัดการเหตุฉุกเฉิน (Incident Management) ขาดความชัดเจน ซึ่งอาจส่งผลต่อการปฏิบัติตามมาตรฐาน ISO/IEC 27001 หรือมาตรฐานการให้บริการ IT อื่น ๆ 4. **สรุปความเสี่ยง** - **ความเสี่ยงด้านการให้บริการ (Service Level)**: เวลาตอบสนองช้าลง 2 ชั่วโมงและไม่มีการกำหนดเวลาการแก้ไขเหตุการณ์วิกฤตอย่างชัดเจน ทำให้ความเสี่ยงต่อการหยุดชะงักของระบบเพิ่มขึ้น - **ความเสี่ยงด้านความปลอดภัย (Security)**: การลบบริการอัปเดตแพตช์และการตรวจสอบความปลอดภัยจากข้อความใหม่ ทำให้ระบบอาจไม่ได้รับการอัปเดตที่จำเป็นในเวลาที่เหมาะสม ส่งผลให้ความเสี่ยงต่อช่องโหว่เพิ่มขึ้น - **ความเสี่ยงด้านการปฏิบัติตามสัญญา (Contractual)**: ขอบเขตการให้บริการที่ลดลงอาจทำให้ข้อกำหนดในสัญญาเดิมไม่สอดคล้องกับเอกสาร SOW ใหม่ จำเป็นต้องมีการปรับสัญญาหรือทำข้อตกลงเสริมเพื่อหลีกเลี่ยงข้อพิพาทในอนาคต **ข้อแนะนำ**: ควรจัดทำเอกสารเสริม (Addendum) เพื่อระบุ SLA ใหม่อย่างชัดเจน, กำหนดกระบวนการแจ้งเหตุฉุกเฉินและช่องทางการติดต่อ, รวมถึงกำหนดขอบเขตการให้บริการ Data Engineer อย่างละเอียด (เช่น การอัปเดตแพตช์ความปลอดภัย) หากลูกค้าต้องการรักษาระดับการบริการเดิม1. มุมมองลูกค้า - ตรวจสอบว่าการเพิ่ม Response Time จาก 2 ชม. เป็น 4 ชม. ตรงกับความต้องการของธุรกิจหรือไม่ หากไม่พอควรต่อรองให้ผู้ให้บริการคืน SLA เดิมหรือเสนอชดเชย (เช่น Service Credit) เพื่อบรรเทาความเสียหายจากการตอบสนองช้าลง - ขอให้ผู้ให้บริการจัดทำ Addendum ระบุขั้นตอนและช่องทางการแจ้งเหตุฉุกเฉิน (รวมหมายเลขโทรศัพท์หรือระบบ Ticket) อย่างชัดเจน เพื่อไม่ให้เกิดความล่าช้าในการจัดการ Critical Incident - ตรวจสอบว่าการลดขอบเขต Data Engineer (เหลือเพียง 5 Manday) ครอบคลุมความต้องการด้านการอัปเดตแพตช์ ความปลอดภัย และการอัปเดต LLMs หรือไม่ หากไม่ควรขอเพิ่มบริการหรือทำสัญญาเสริมเพื่อรักษาระดับความปลอดภัยของระบบ - ประเมินความเสี่ยงต่อการหยุดทำงานและช่องโหว่ด้านความปลอดภัยจากการลบ SLA ของ Critical Incident และบริการอัปเดตแพตช์; หากความเสี่ยงสูงควรกำหนดมาตรการสำรอง (เช่น การใช้ผู้ให้บริการอื่นหรือบริการ Managed Security) 2. มุมมองผู้ให้บริการ - จัดทำ Addendum อย่างเป็นทางการเพื่อบันทึกการเปลี่ยนแปลง SLA (Response Time 4 ชม.) และอธิบายเหตุผลของการเปลี่ยนแปลง เพื่อให้สอดคล้องกับกฎหมายและมาตรฐาน ISO/IEC 27001 - กำหนดกระบวนการแจ้งเหตุฉุกเฉินใหม่ (ช่องทางติดต่อ, เวลาตอบสนอง, ระยะเวลาการแก้ไข) และสื่อสารให้ลูกค้าทราบอย่างชัดเจน; ควรเพิ่มช่องทางดิจิทัล (e‑mail, portal) หากไม่มีหมายเลขโทรศัพท์ - พิจารณาให้บริการ Data Engineer เพิ่มเติมหรือเสนอแพคเกจเสริม (เช่น Patch Management, Security Update) เพื่อรักษามาตรฐานความปลอดภัยและตอบสนองความต้องการของลูกค้า; ระบุขอบเขตและค่าใช้จ่ายอย่างโปร่งใส - ทบทวนผลกระทบต่อสัญญาเดิมและทำการปรับปรุงข้อกำหนดให้สอดคล้องกับการลดขอบเขตบริการ; ตรวจสอบว่าการเปลี่ยนแปลงไม่ขัดต่อข้อผูกพันเดิมหรือกฎหมายคุ้มครองผู้บริโภค - รายละเอียดการ Backup แบบ Retention (Docker config, Docker Image, Docker Volume, LocalLLM Model) และการ Backup แบบ Internal Backup รวมถึงสคริปต์ auto‑delete, การแจ้งเตือนอีเมล, การจัดการ External backup โดย VSTECS - รายละเอียดการ Restore จาก Internal Backup, การใช้ External backup เมื่อ Internal backup ไม่สามารถใช้งานได้, การทำ Full Restore ของ Data Volume, ผลกระทบต่อ Compose/Image/LLM, การจัดการ Nvidia driver, RTO (Recovery Time Objective) และ RPO (Recovery Point Objective) รวมถึงการกำหนดระยะเวลา 1 วันและ 24 ชั่วโมงตามลำดับ - การจัดทำวิธีการ monitoring และการแจ้งเตือน (CPU, Memory, Disk Threshold, Backup/Restore success‑fail) โดย VSTECS จัดเตรียม SMTP Server การลบข้อมูลเหล่านี้หมายความว่าข้อกำหนดและความรับผิดชอบของลูกค้าในเรื่องการสำรองและกู้คืนระบบ AI Knowledge Management (KM‑AI) ไม่ได้ถูกระบุไว้ในสัญญา/ข้อตกลงฉบับใหม่ ซึ่งอาจทำให้เกิดความไม่ชัดเจนเกี่ยวกับระดับการให้บริการ (SLA), ความเสี่ยงต่อการสูญเสียข้อมูล, และความรับผิดชอบของผู้ให้บริการต่อการกู้คืนระบบในกรณีเหตุฉุกเฉิน 2. **การแก้ไขหัวข้อและรูปแบบ** – หัวข้อ "4. ควำมรับผิดชอบของลูกค้ำ" ถูกแก้เป็น "4. ความรับผิดชอบของลูกค้า" (การแก้ไขการสะกดและการใช้คำ) และการจัดรูปแบบของรายการ bullet ถูกทำให้สั้นลงโดยลบหมายเลข "4.1" และทำให้รายการอยู่ในรูปแบบย่อเดียวกัน ทั้งนี้เป็นการแก้ไขที่ไม่ส่งผลต่อความเสี่ยงหรือสาระสำคัญของเนื้อหา 3. **การลบส่วนซ้ำซ้อน** – มีการลบส่วนที่ซ้ำกันของหัวข้อ "4.3 Restore KM‑AI System" ที่ปรากฏสองครั้งในข้อความเดิม ทำให้ข้อความใหม่มีความกระชับมากขึ้น แต่เนื้อหาที่สำคัญเกี่ยวกับการ Restore ถูกลบไปทั้งหมดเช่นเดียวกัน สรุป: การเปลี่ยนแปลงที่สำคัญคือการ **ลบข้อมูลเกี่ยวกับการสำรองและกู้คืนระบบ** ซึ่งเป็นส่วนที่มีผลต่อความรับผิดชอบของลูกค้าและระดับการให้บริการของผู้ให้บริการ หากไม่มีการระบุข้อกำหนดเหล่านี้ในสัญญาใหม่ อาจทำให้เกิดความเสี่ยงด้านการจัดการข้อมูลและการฟื้นฟูระบบในกรณีเกิดเหตุฉุกเฉินการลบส่วนที่เกี่ยวกับการสำรองและกู้คืนระบบ (Backup/Restore KM‑AI) จากสัญญาใหม่ส่งผลต่อความหมายและความรับผิดชอบของทั้งสองฝ่ายอย่างชัดเจน ดังนั้นควรดำเนินการดังต่อไปนี้: 1. มุมมองของลูกค้า (ผู้ได้รับบริการ) - ตรวจสอบว่าตนยังคงมีความคุ้มครองต่อความเสี่ยงการสูญเสียข้อมูลหรือไม่; หากไม่มีให้เรียกร้องให้ผู้ให้บริการเพิ่มข้อกำหนด Backup/Restore เข้าไปในสัญญาใหม่หรือจัดทำข้อตกลงเสริม (Addendum) ที่ระบุ RTO, RPO, ระยะเวลาการเก็บรักษาข้อมูล, กระบวนการตรวจสอบความสำเร็จของ Backup/Restore และการแจ้งเตือน. - ขอให้ผู้ให้บริการจัดทำแผนการกู้คืนฉุกเฉิน (Disaster Recovery Plan) ที่ชัดเจน รวมถึงขั้นตอนการทดสอบ (DR Drill) อย่างน้อยปีละหนึ่งครั้ง เพื่อยืนยันความพร้อมของระบบ. - กำหนดความรับผิดชอบต่อค่าใช้จ่ายที่อาจเกิดจากการกู้คืนข้อมูล (เช่น ค่าจ้าง VSTECS, ค่าซื้อ License เพิ่ม) ให้ระบุในสัญญาอย่างชัดเจน เพื่อหลีกเลี่ยงค่าใช้จ่ายที่ไม่คาดคิด. - ตรวจสอบว่าผู้ให้บริการมีการจัดเตรียมเครื่องมือมอนิเตอร์และการแจ้งเตือน (CPU, Memory, Disk, Backup/Restore success‑fail) พร้อม SMTP Server หรือไม่; หากไม่มีให้เพิ่มข้อกำหนดนี้เป็นส่วนหนึ่งของ SLA. 2. มุมมองของผู้ให้บริการ (ผู้ขาย) - ประเมินความเสี่ยงด้านกฎหมายและความรับผิดชอบต่อการสูญเสียข้อมูลของลูกค้าเมื่อไม่มีข้อกำหนด Backup/Restore ในสัญญา; ควรพิจารณาเพิ่มข้อยกเว้นหรือข้อจำกัดความรับผิดชอบ (Limitation of Liability) ที่สอดคล้องกับมาตรฐานอุตสาหกรรม (เช่น ISO/IEC 27001) เพื่อปกป้องตนเอง. - จัดทำเอกสารแนบ (Annex) ที่อธิบายบริการ Backup/Restore ที่มีอยู่ (Internal Backup, External Backup ผ่าน VSTECS) พร้อมรายละเอียดเทคนิค (Docker config, Image, Volume, LocalLLM Model) และขั้นตอนการกู้คืนเต็มรูปแบบ; ให้ลูกค้าเลือกใช้เป็นบริการเสริม (Optional Service) หากต้องการ. - กำหนด SLA ที่ชัดเจนสำหรับบริการ Backup/Restore (เช่น RPO ≤ 24 ชั่วโมง, RTO ≤ 1 วัน) หากลูกค้าเลือกใช้บริการเสริม; ระบุค่าใช้จ่ายเพิ่มเติมและเงื่อนไขการยกเลิก. - จัดทำกระบวนการตรวจสอบและรายงานผลการสำรองข้อมูลเป็นประจำ (เช่น รายงานรายสัปดาห์) พร้อมการแจ้งเตือนอีเมลตามที่ลูกค้าต้องการ; เพื่อแสดงความโปร่งใสและลดข้อโต้แย้งในอนาคต. สรุป: ควรเพิ่มหรือแนบข้อกำหนด Backup/Restore เข้าไปในสัญญาใหม่ หรือทำข้อตกลงเสริมที่ระบุรายละเอียดเทคนิค, SLA, ค่าใช้จ่ายและความรับผิดชอบอย่างชัดเจน ทั้งสองฝ่ายจะได้ลดความเสี่ยงด้านการสูญเสียข้อมูลและความขัดแย้งในภายหลังpage 5MODIFIED
