สรุปภาพรวม

โดยรวมมีการเปลี่ยนแปลงจำนวน 2 รายการ (เพิ่ม 0 รายการ, ลบ 0 รายการ, แก้ไข 2 รายการ)

ส่วนที่ 1 – สรุปภาพรวมการเปลี่ยนแปลง
- หน้า 2: ระยะเวลา Resolution Time (Work‑around) สำหรับ “Server down ที่ไม่ใช่ปัญหาจาก Hardware” ถูกปรับจาก 4 ชั่วโมงเป็น 1 วัน; ส่วน “Service Down” ยังคง 4 ชั่วโมง (การแก้ไขเป็นการจัดรูปแบบและการสะกดคำเท่านั้น)
- หน้า 5: Recovery Time Objective (RTO) ของ Critical Incident ถูกขยายจาก 4 ชั่วโมงเป็น 1 วัน และข้อความอธิบาย “OS 4 ชั่วโมงพอได้” ถูกลบออกทั้งหมด

ส่วนที่ 2 – ข้อเสนอแนะภาพรวม

1) มุมมองผู้ได้รับบริการ (ลูกค้า)
- ตรวจสอบสัญญา SLA ปัจจุบันว่ามีการกำหนดเวลา 4 ชั่วโมงหรือไม่; หากมีให้ดำเนินการเพิ่ม Addendum เพื่อบันทึกเวลาใหม่เป็น 1 วันและรับลายเซ็นยืนยัน
- ประเมินผลกระทบต่อการดำเนินงาน (downtime เพิ่ม, ความเสียหายต่อธุรกิจ) และพิจารณาขอชดเชยหรือส่วนลดตามข้อตกลง SLA
- จัดทำแผนสำรอง (fallback) เช่น ระบบสำรองหรือการกระจายโหลด เพื่อลดความเสี่ยงจาก downtime ที่ยาวนานขึ้น
- บันทึกและติดตามการตอบสนองของผู้ให้บริการเป็นหลักฐานสำหรับการเจรจาหรือดำเนินการทางกฎหมาย

2) มุมมองผู้ให้บริการ (ผู้ขาย)
- ยืนยันว่าการปรับ SLA ได้รับการอนุมัติจากฝ่ายกฎหมายและผู้บริหาร; จัดทำ Addendum/Amendment พร้อมลายมือชื่อของทั้งสองฝ่าย
- แจ้งลูกค้าอย่างเป็นทางการล่วงหน้าเกี่ยวกับการเปลี่ยนแปลงเวลา Resolution Time และ RTO พร้อมอธิบายเหตุผลและผลกระทบที่คาดว่าจะเกิดขึ้น
- ปรับกระบวนการ Incident Management และ Recovery Plan ให้สอดคล้องกับระยะเวลาใหม่ (Work‑around ≤ 1 วัน, RTO ≤ 1 วัน) และทดสอบ DR อย่างสม่ำเสมอ
- ปรับเอกสาร SLA, คู่มือบริการ, และเงื่อนไขการชำระเงินให้สอดคล้องกับระดับบริการที่ลดลง

ส่วนที่ 3 – ผู้มีส่วนได้ส่วนเสียและผลกระทบ

- ลูกค้า (ผู้รับบริการ): จะเผชิญกับระยะเวลาการแก้ไขและฟื้นฟูที่ยาวนานขึ้น ส่งผลต่อการดำเนินงานและอาจต้องจัดสรรทรัพยากรสำรองเพิ่มขึ้น
- ผู้ให้บริการ (ผู้ขาย): ต้องปรับกระบวนการภายใน, ปรับสัญญา, และอาจต้องรับข้อเรียกร้องค่าปรับหรือการชดเชยจากลูกค้า
- ทีมปฏิบัติการด้าน IT ของลูกค้า: ต้องอัปเดตแผน BCP/DR และฝึกอบรมการตอบสนองต่อเหตุการณ์ตามระยะเวลาใหม่
- ฝ่ายกฎหมายของทั้งสององค์กร: ต้องตรวจสอบและจัดทำเอกสารสัญญาเพิ่มเติม (Addendum/Amendment) เพื่อให้การเปลี่ยนแปลงมีผลบังคับใช้
- ผู้ใช้ปลายทางของลูกค้า: อาจประสบกับการหยุดให้บริการที่ยาวนานขึ้น ส่งผลต่อความพึงพอใจและการใช้งานระบบโดยตรง.

2. ขอบเขตกำรให้บริกำร
การให้บริการภายใต้ SOW นี้ประกอบด้วย:
2.1 กำรออกแบบและติดตั้งระบบ
2.1.1 การออกแบบและกำหนดค่าระบบ
- วิเคราะห์ความต้องการของลูกค้าและออกแบบระบบ Softnix GenAI ให้เหมาะสม
- กำหนดค่าระบบตามแผนการออกแบบที่ได้รับการอนุมัติ
2.1.2 การตั้งค่า AI Agent Services
- ปรับแต่ง AI Agent Services เพื่อรองรับการใช้งานเฉพาะของลูกค้า โดยมีรายละเอียดดังนี้:
a) การปรับปรุงข้อมูล:
- เตรียมและปรับปรุงข้อมูลสำหรับ AI Knowledge จำนวนไม่เกิน 10 ชุดข้อมูล
- ทำความสะอาดและจัดรูปแบบข้อมูลให้เหมาะสมสำหรับการใช้งานกับ AI
b) การออกแบบ Prompt Instruction:
- สร้างและปรับแต่งคำสั่งและคำแนะนำสำหรับ AI ให้เหมาะสมกับลักษณะการใช้งานของ
ลูกค้า
- ทดสอบและปรับปรุง Prompt เพื่อให้ได้ผลลัพธ์ที่แม่นยำและตรงตามความต้องการ
c) การเลือกและตั้งค่า LLM:
- คัดเลือก Large Language Model (LLM) ที่เหมาะสมกับความต้องการของลูกค้า (รายการ
LLM ที่ลูกค้าใช้ได้จะขึ้นอยู่กับรุ่นของสินค้า)
- ปรับแต่งพารามิเตอร์ของ LLM เพื่อให้ได้ประสิทธิภาพสูงสุด
d) การเชื่อมต่อกับระบบภายนอก:
- ดำเนินการเชื่อมต่อกับ Line Services (ลูกค้าเป็นผู้จัดหา Line Services)
- ทดสอบการเชื่อมต่อและแก้ไขปัญหาที่อาจเกิดขึ้น
e) การติดตั้ง Web Chat Application:
- ติดตั้ง Web Chat Application บนระบบของลูกค้า
- หรือ ผสานรวม Web Chat เข้ากับ Web Application ที่มีอยู่ของลูกค้า
- ทดสอบการทำงานและปรับแต่งตามความต้องการ
2.2 กำรฝึกอบรม
2.2.1 การอบรมการใช้งานระบบ ระยะเวลา 1 วัน โดยครอบคลุมเนื้อหาดังต่อไปนี้
- ครอบคลุมหัวข้อ: การใช้งานพื้นฐาน, การจัดการผู้ใช้, การตั้งค่าระบบ, และการแก้ไขปัญหา

ตัวอย่างของ Change Request ที่อาจเกิดขึ้นในโครงการ:		### การเปลี่ยนแปลงที่สำคัญระหว่างข้อความเดิมและข้อความใหม่ 1. **ส่วน 2.3.1 การให้คำปรึกษาออนไลน์ – ระยะเวลา Resolution Time (Workaround) สำหรับ Server down** - **ข้อความเดิม**: ระบุว่า "Resolution Time (Workaround) : ภายใน 4 ชั่วโมง (เบิ้มแจ้งลค 1 วัน)" และต่อด้วย "Service Down ใช้เวลา 4 ชั่วโมง". - **ข้อความใหม่**: ยกเลิกการระบุเวลา 4 ชั่วโมงสำหรับ Server down และแทนที่ด้วยรายการย่อยใหม่: - "Server down ที่ไม่ใช่ปัญหาจาก Hardware : 1 วัน" - "Service Down ใช้เวลา 4 ชั่วโมง" (ยังคงอยู่เดิม) - **ผลกระทบ**: การเปลี่ยนแปลง SLA นี้ทำให้ระยะเวลาการแก้ไข (Workaround) สำหรับ Server down ที่ไม่เกี่ยวกับฮาร์ดแวร์เพิ่มจาก 4 ชั่วโมงเป็น 1 วัน ซึ่งเป็นการลดระดับการให้บริการ (downgrade) ของระดับความเร่งด่วน. หากสัญญาเดิมหรือข้อกำหนดของลูกค้ากำหนดให้ต้องตอบสนองภายใน 4 ชั่วโมง การเปลี่ยนแปลงนี้อาจทำให้เกิดความเสี่ยงต่อการละเมิดสัญญาและอาจกระตุ้นการเรียกร้องค่าปรับหรือการยกเลิกสัญญาได้. - **ข้อควรพิจารณา**: ควรตรวจสอบว่ามีการอัปเดตข้อกำหนด SLA ในสัญญา (เช่น การแก้ไข Annex หรือ Addendum) อย่างเป็นทางการหรือไม่, และต้องสื่อสารให้ลูกค้าทราบและรับความยินยอมเป็นลายลักษณ์อักษรก่อนการบังคับใช้. 2. **ส่วน 2.3.2 บริการ Data Engineer สำหรับ Change Request** - การเปลี่ยนแปลงหลักเป็นเพียงการแก้ไขการสะกดคำบางคำ (เช่น "อับเดท" แทน "อัปเดท") ซึ่งเป็นการแก้ไขเชิงรูปแบบ/การพิมพ์เท่านั้น. ตามเงื่อนไขพิเศษของคุณ, การแก้ไขเชิงรูปแบบที่ไม่มีผลต่อสาระสำคัญ **ไม่ต้องระบุความเสี่ยงที่มีนัยสำคัญ**. 3. **ส่วนอื่น ๆ ของเอกสาร** - ส่วนอื่น ๆ ของข้อความ (2.1, 2.1.1, 2.1.2, 2.2, รายการของ Prompt Engineering, การเชื่อมต่อ Line Services, การติดตั้ง Web Chat Application ฯลฯ) ไม่พบการเพิ่มหรือการลบข้อมูลสาระสำคัญใด ๆ. การเปลี่ยนแปลงที่เกิดขึ้นเป็นเพียงการจัดรูปแบบหรือการพิมพ์ซ้ำซ้อนที่ไม่มีผลต่อขอบเขตงาน. ### สรุปความเสี่ยงโดยรวม - **ความเสี่ยงหลัก** อยู่ที่การเปลี่ยนแปลง SLA ของ Server down (จาก 4 ชั่วโมงเป็น 1 วัน) ซึ่งอาจทำให้เกิดความขัดแย้งกับข้อกำหนดสัญญาเดิมและอาจส่งผลต่อการบังคับใช้ค่าปรับหรือการพิจารณาความพึงพอใจของลูกค้า. - **ความเสี่ยงรอง** จากการแก้ไขการสะกดคำหรือการจัดรูปแบบไม่มีนัยสำคัญต่อข้อผูกมัดทางกฎหมายหรือการเงิน. **ข้อแนะนำ**: หากการเปลี่ยนแปลง SLA นี้เป็นที่ต้องการของผู้ให้บริการ ควรทำการแก้ไขสัญญาอย่างเป็นทางการ (Addendum) พร้อมรับลายเซ็นจากลูกค้า, ปรับปรุงเอกสาร Service Level Agreement (SLA) ให้สอดคล้อง, และอัปเดตกระบวนการจัดการเหตุฉุกเฉิน (Incident Management) ให้สอดคล้องกับระยะเวลาใหม่.1. มุมมองลูกค้า: • ตรวจสอบสัญญาเดิมว่ามีการกำหนดระยะเวลาแก้ไข Server down อย่างชัดเจนหรือไม่ หากมีให้เรียกร้องให้ผู้ให้บริการแก้ไขสัญญาโดยเพิ่ม Addendum ที่ระบุเวลา 1 วันและขอรับการยอมรับเป็นลายลักษณ์อักษร • ประเมินผลกระทบต่อการดำเนินงานขององค์กร เช่น ความล่าช้าในการให้บริการต่อผู้ใช้ปลายทาง หากผลกระทบสำคัญให้พิจารณาต่อรองค่าปรับหรือขอส่วนลดตามข้อตกลง SLA • จัดทำแผนสำรอง (fallback) เช่น ระบบสำรองหรือการกระจายโหลด เพื่อลดความเสี่ยงจากการ downtime ที่ยาวนานขึ้น • ติดตามและบันทึกการตอบสนองของผู้ให้บริการอย่างเป็นระบบ เพื่อใช้เป็นหลักฐานในการเจรจาหรือดำเนินการทางกฎหมายหากเกิดการละเมิดสัญญา 2. มุมมองผู้ให้บริการ: • ตรวจสอบว่าการเปลี่ยนแปลง SLA ได้รับการอนุมัติจากฝ่ายกฎหมายและผู้บริหารแล้วหรือยัง และจัดทำ Addendum หรือ Amendment ที่ลงลายมือชื่อของทั้งสองฝ่ายเพื่อให้การเปลี่ยนแปลงมีผลบังคับใช้ • แจ้งลูกค้าอย่างเป็นทางการล่วงหน้าเกี่ยวกับการปรับเปลี่ยนระยะเวลาแก้ไข Server down ที่ไม่ใช่ฮาร์ดแวร์ พร้อมอธิบายเหตุผลและผลกระทบที่คาดว่าจะเกิดขึ้น เพื่อให้ลูกค้าเข้าใจและยอมรับ • ปรับปรุงกระบวนการ Incident Management ให้สอดคล้องกับระยะเวลาใหม่ เช่น กำหนดขั้นตอน Workaround ภายใน 1 วัน, จัดสรรทรัพยากรเพิ่มเติม หรือใช้เครื่องมืออัตโนมัติเพื่อลดเวลาการตอบสนอง • ทบทวนและอัปเดตเอกสาร SLA, คู่มือบริการ, และสัญญาเพื่อให้ข้อมูลสอดคล้องกันทั้งหมด • เตรียมพร้อมรับข้อร้องเรียนหรือการเรียกร้องค่าปรับจากลูกค้าโดยมีเอกสารสนับสนุนการเปลี่ยนแปลงที่เป็นทางการpage 2MODIFIED
4. ควำมรับผิดชอบของลูกค้ำ
4.1 เพื่อให้การดำเนินโครงการเป็นไปอย่างราบรื่น ลูกค้าจะต้องรับผิดชอบในเรื่องต่อไปนี้:
- จัดเตรียมข้อมูลที่จำเป็นสำหรับการสร้าง AI Knowledge
- จัดหาและให้สิทธิ์การเข้าถึงระบบที่เกี่ยวข้อง เช่น Line Services
- จัดสรรบุคลากรที่เกี่ยวข้องเพื่อเข้าร่วมการฝึกอบรมและให้ข้อมูลที่จำเป็น
- แจ้งปัญหาหรือข้อขัดข้องให้ทีมบริการทราบโดยเร็วที่สุด

• การ Backup แบบ Retention ข้อมูลดังนี้

• Docker Volume 30 Day
• LocalLLM Model 3 Copy

• อ้างอิง Retention จากข้อด้านบน
• Script auto delete หรือ เขียนทับเมื่อครบ กำหนด retention 30 วัน
• Alert email กรณี Back up Success หรือ Fail
• External backup ทาง VSTECS จะเป็นผู้ดูแลโดย จะไป backup ตาม Destination

4.3 Restore KM-AI System

• การใช้ External backup ใช้เมื่อ ไม่สามารถใช้งานข้อมูลจาก Internal Backup ได้
• การ Restore ในส่วนของ Data (Volume) จะเป็น แบบ Full Restore เป็นผลให้ข้อมูล
ที่เพิ่มเข้าไปหลังการ backup จะหายทั้งหมด
• การ Restore ระดับ compose, image, LLM จะไม่กระทบกับ Data Volume
• Nvidia driver ในกรณีที่มีการเปลี่ยนการ์ด เนื่องจาก Hardware
• เสีย Softnix จะเป็นผู้ระบุ รุ่นการ์ด ที่ Support ที่ทำให้ OS มองเห็นหรือใช้งานได้
โดย VSTECS จะเป็นผู้จัดหา และทาง Softnix จะเป็นผู้ดำเนินการติดตั้ง Nvidia
driver จนสามรถกลับมาใช้งานได้
• Recovery Time(RTO) ในกรณีเครื่อง เสียหาย จำเป็นต้อง Install และ โดยเริ่ม
หลังจากที่ HPE เปลี่ยน Hardware เรียบร้อยภายใน4 ชั่วโมง ของ Critical incident
(ขอเปลี่ยนเป็น1วัน) OS 4 ชั่วโมงพอได้

ควรมีการอัปเดตข้อตกลงและสื่อสารให้ลูกค้าทราบอย่างชัดเจนเพื่อหลีกเลี่ยงข้อขัดแย้งในอนาคต. - ส่วนอื่นของข้อความ (หัวข้อ 4.1, 4.2, รายการ Backup, รายการ Restore อื่น ๆ) ไม่พบการเปลี่ยนแปลงสาระสำคัญและไม่มีความเสี่ยงที่มีนัยสำคัญจากการแก้ไขนี้.1. มุมมองลูกค้า (ผู้รับบริการ) - ตรวจสอบสัญญา SLA ปัจจุบันว่ามีการระบุ RTO เป็น 4 ชั่วโมงหรือไม่ หากมีต้องทำการแก้ไขสัญญาให้สอดคล้องกับ RTO ใหม่ 1 วัน และขอรับการยืนยันเป็นลายลักษณ์อักษรจากผู้ให้บริการ - ประเมินผลกระทบต่อการดำเนินธุรกิจของตน เช่น ความเสียหายจากการหยุดทำงาน 1 วัน เปรียบเทียบกับค่าเสียหายที่ยอมรับได้ หากผลกระทบเกินกว่าที่ยอมรับได้ควรต่อรองให้มีการชดเชยหรือปรับเงื่อนไขอื่น (เช่น การให้เครดิต, บริการเสริม) - แจ้งผู้ใช้ปลายทางหรือหน่วยงานที่เกี่ยวข้องล่วงหน้าเกี่ยวกับการเปลี่ยนแปลง RTO เพื่อให้มีการวางแผนการทำงานสำรองหรือการจัดการความต่อเนื่องของธุรกิจ (BCP) อย่างเหมาะสม 2. มุมมองผู้ให้บริการ (ผู้ขาย) - ปรับเอกสาร SLA, ข้อกำหนดและเงื่อนไข (Terms & Conditions) ให้ระบุ RTO ใหม่เป็น 1 วันอย่างชัดเจน และลบข้อความอ้างอิงเวลา OS 4 ชั่วโมงออกทั้งหมด - จัดทำแผนฟื้นฟูระบบ (Recovery Plan) ที่สอดคล้องกับ RTO 1 วัน รวมถึงการจัดสรรทรัพยากร, การทดสอบ DR (Disaster Recovery) อย่างสม่ำเสมอเพื่อให้มั่นใจว่าสามารถคืนระบบได้ภายในระยะเวลาที่กำหนด - สื่อสารการเปลี่ยนแปลงนี้ต่อผู้ว่าจ้างอย่างเป็นทางการ (เช่น จดหมายแจ้งเปลี่ยนแปลง SLA) พร้อมอธิบายเหตุผลและผลกระทบที่อาจเกิดขึ้น รวมถึงข้อเสนอการชดเชยหรือบริการเพิ่มเติมเพื่อรักษาความพึงพอใจของลูกค้า - พิจารณาการปรับค่าใช้จ่ายหรือเงื่อนไขการชำระเงินให้สอดคล้องกับระดับบริการใหม่ หาก RTO ยาวนานกว่าเดิมอาจต้องมีการทบทวนค่าใช้จ่ายด้านการสนับสนุนและการบำรุงรักษาpage 5MODIFIED
