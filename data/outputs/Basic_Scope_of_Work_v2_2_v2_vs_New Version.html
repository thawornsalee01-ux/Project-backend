สรุปภาพรวม

Overall Risk Level: MEDIUM

โดยรวมมีการเปลี่ยนแปลงจำนวน 2 รายการ (เพิ่ม 0 รายการ, ลบ 0 รายการ, แก้ไข 2 รายการ)

ส่วนที่ 1 : สรุปภาพรวมการเปลี่ยนแปลง
- Page 2 : ปรับ “Resolution Time (Workaround)” สำหรับกรณี Server down ที่ไม่ใช่ปัญหา Hardware จาก 1 วันเป็น 4 ชั่วโมง (หมายเหตุ “เบิ้มแจ้งลค 1 วัน”) – ผลกระทบต่อ SLA อย่างมีนัยสำคัญ; ส่วนอื่นเป็นการจัดรูปแบบหรือการพิมพ์ที่ไม่ส่งผลต่อสาระสำคัญ.
- Page 5 : แก้ไขข้อความ RTO ในหัวข้อ 4.3 Restore KM‑AI System จาก “ติดตั้งใหม่ภายใน 1 วันหลังจาก HPE เปลี่ยนฮาร์ดแวร์” เป็น “ภายใน 4 ชั่วโมงของ Critical incident (ขอเปลี่ยนเป็น 1 วัน) OS 4 ชั่วโมงพอได้” – ทำให้ระยะเวลา RTO ไม่ชัดเจนและเพิ่มเงื่อนไขใหม่.

ส่วนที่ 2 : สรุปข้อเสนอแนะภาพรวม

1) มุมมองผู้ได้รับบริการ (ลูกค้า)
- ตรวจสอบว่า SLA ใหม่ (Resolution Time 4 ชม.) สอดคล้องกับความต้องการและความสามารถขององค์กรหรือไม่; หากไม่ตรงควรเจรจาต่อรอง.
- ขอให้ผู้ให้บริการชี้แจง RTO ที่แน่นอน (4 ชม. หรือ 1 วัน) พร้อมกำหนดนิยามของ “Critical incident” อย่างชัดเจนในสัญญา.
- เฝ้าระวังความเสี่ยงต่อการละเมิด SLA/RTO ที่อาจทำให้ต้องชดเชยหรือส่งผลต่อแผนต่อเนื่องธุรกิจ (BCP) และเตรียมมาตรการสำรองข้อมูล/ระบบให้สอดคล้องกับ RTO ที่สั้นลง.

2) มุมมองผู้ให้บริการ (ผู้ขาย)
- ปรับปรุงและบันทึก SLA/RTO อย่างเป็นทางการในเอกสารสัญญา โดยระบุชัดเจนว่า “Critical incident” หมายถึงอะไรและต้องฟื้นฟูภายในกี่ชั่วโมง.
- ป้องกันความเข้าใจคลาดเคลื่อนโดยลบข้อความที่ขัดแย้ง (4 ชม. vs 1 วัน) และให้คำอธิบายเพิ่มเติมในหมายเหตุหรือภาคผนวก.
- จัดทำเอกสารสนับสนุน เช่น คู่มือการตอบสนอง, รายงานการทดสอบ DR, บันทึกการฝึกอบรมทีม 24/7, และขั้นตอนการชดเชย/ค่าปรับกรณีไม่บรรลุ SLA/RTO.  ✅  ✅

2. ขอบเขตกำรให้บริกำร
การให้บริการภายใต้ SOW นี้ประกอบด้วย:
2.1 กำรออกแบบและติดตั้งระบบ
2.1.1 การออกแบบและกำหนดค่าระบบ
- วิเคราะห์ความต้องการของลูกค้าและออกแบบระบบ Softnix GenAI ให้เหมาะสม
- กำหนดค่าระบบตามแผนการออกแบบที่ได้รับการอนุมัติ
2.1.2 การตั้งค่า AI Agent Services
- ปรับแต่ง AI Agent Services เพื่อรองรับการใช้งานเฉพาะของลูกค้า โดยมีรายละเอียดดังนี้:
a) การปรับปรุงข้อมูล:
- เตรียมและปรับปรุงข้อมูลสำหรับ AI Knowledge จำนวนไม่เกิน 10 ชุดข้อมูล
- ทำความสะอาดและจัดรูปแบบข้อมูลให้เหมาะสมสำหรับการใช้งานกับ AI
b) การออกแบบ Prompt Instruction:
- สร้างและปรับแต่งคำสั่งและคำแนะนำสำหรับ AI ให้เหมาะสมกับลักษณะการใช้งานของ
ลูกค้า
- ทดสอบและปรับปรุง Prompt เพื่อให้ได้ผลลัพธ์ที่แม่นยำและตรงตามความต้องการ
c) การเลือกและตั้งค่า LLM:
- คัดเลือก Large Language Model (LLM) ที่เหมาะสมกับความต้องการของลูกค้า (รายการ
LLM ที่ลูกค้าใช้ได้จะขึ้นอยู่กับรุ่นของสินค้า)
- ปรับแต่งพารามิเตอร์ของ LLM เพื่อให้ได้ประสิทธิภาพสูงสุด
d) การเชื่อมต่อกับระบบภายนอก:
- ดำเนินการเชื่อมต่อกับ Line Services (ลูกค้าเป็นผู้จัดหา Line Services)
- ทดสอบการเชื่อมต่อและแก้ไขปัญหาที่อาจเกิดขึ้น
e) การติดตั้ง Web Chat Application:
- ติดตั้ง Web Chat Application บนระบบของลูกค้า
- หรือ ผสานรวม Web Chat เข้ากับ Web Application ที่มีอยู่ของลูกค้า
- ทดสอบการทำงานและปรับแต่งตามความต้องการ
2.2 กำรฝึกอบรม
2.2.1 การอบรมการใช้งานระบบ ระยะเวลา 1 วัน โดยครอบคลุมเนื้อหาดังต่อไปนี้
- ครอบคลุมหัวข้อ: การใช้งานพื้นฐาน, การจัดการผู้ใช้, การตั้งค่าระบบ, และการแก้ไขปัญหา
เบื้องต้น, หลักการของ Prompt Engineering, เทคนิคการเขียน Prompt ที่มีประสิทธิภาพ, การ
ปรับแต่ง Prompt เพื่อผลลัพธ์ที่ดีขึ้น

การลดระยะเวลา Resolution Time สำหรับ Server down จาก 1 วันเป็น 4 ชั่วโมงเพิ่มความเสี่ยงด้านกฎหมายอย่างชัดเจน เนื่องจากอาจทำให้ผู้ให้บริการต้องเผชิญกับข้อเรียกร้องหรือการฟ้องร้องจากลูกค้า หากไม่สามารถตอบสนองตาม SLA ที่เข้มงวดนี้ได้ อีกทั้งความเสี่ยงด้านการเงินก็เพิ่มขึ้นจากความเป็นไปได้ของการต้องจ่ายค่าปรับหรือค่าเสียหายตามสัญญาเมื่อไม่สามารถทำให้ระบบกลับมาทำงานได้ภายในเวลาที่กำหนด ส่วนความเสี่ยงด้านการดำเนินงานก็สูงขึ้น เนื่องจากทีมงานต้องเร่งตอบสนองและแก้ไขปัญหาในระยะเวลาที่สั้นลง ซึ่งอาจกดดันทรัพยากรและกระบวนการทำงานเดิม ทำให้เกิดความล่าช้าในงานอื่น ๆ หรือความผิดพลาดในการแก้ไขปัญหา		page 2MODIFIED

**การวางแผนต่อเนื่องของธุรกิจ**: ลูกค้าอาจต้องปรับแผนการฟื้นฟูธุรกิจ (BCP) ให้สอดคล้องกับ RTO ที่สั้นลง ซึ่งอาจต้องลงทุนเพิ่มในด้านการสำรองข้อมูลหรือระบบสำรองอื่น ๆ **2. มุมมองผู้ให้บริการ / ผู้ขาย** - **ภาระงานเพิ่มขึ้น**: การตั้งเป้าหมายฟื้นฟูภายใน 4 ชั่วโมง (โดยเฉพาะในกรณี Critical incident) ต้องมีทีมสนับสนุนที่พร้อมทำงานตลอด 24/7, กระบวนการอัตโนมัติ, และเครื่องมือที่สามารถทำการกู้คืนได้อย่างรวดเร็ว ซึ่งอาจต้องลงทุนเพิ่มในทรัพยากรและเทคโนโลยี - **ความเสี่ยงต่อการละเมิดสัญญา**: หากไม่สามารถทำตาม RTO ที่ใหม่ได้ ผู้ให้บริการอาจต้องรับผิดชอบต่อค่าปรับหรือการชดเชยตามสัญญา SLA ที่อาจถูกกำหนดให้เข้มงวดกว่าเดิม - **ความสับสนในเอกสารและการสื่อสาร**: การมีข้อความที่ขัดแย้ง (4 ชั่วโมง vs 1 วัน) ทำให้ทีมภายในต้องใช้เวลาในการตีความและอาจทำให้เกิดความล่าช้าในการดำเนินการแก้ไขเหตุฉุกเฉิน ### คำแนะนำเพื่อจัดการความเสี่ยง **สำหรับลูกค้า** 1. ขอให้ผู้ให้บริการชี้แจงและยืนยัน RTO ที่แน่นอน (4 ชั่วโมงหรือ 1 วัน) พร้อมระบุเงื่อนไขของ "Critical incident" อย่างชัดเจน 2. ตรวจสอบสัญญา SLA ปัจจุบันว่ามีการอัปเดตหรือไม่ และหากมีการเปลี่ยนแปลง RTO ต้องทำการแก้ไขสัญญาให้เป็นลายลักษณ์อักษร 3. พิจารณาเพิ่มมาตรการสำรอง (เช่น การทำสำเนาข้อมูลแบบเรียลไทม์) เพื่อให้สอดคล้องกับ RTO ที่สั้นลง 4. จัดทำแผนการสื่อสารฉุกเฉินที่ระบุขั้นตอนและผู้รับผิดชอบในกรณีที่ฟื้นฟูไม่สำเร็จภายในเวลาที่กำหนด **สำหรับผู้ให้บริการ** 1. กำหนดและบันทึก RTO อย่างเป็นทางการในเอกสาร SLA โดยระบุชัดเจนว่า "Critical incident" หมายถึงอะไร และต้องทำการฟื้นฟูภายในกี่ชั่วโมง 2. ปรับปรุงกระบวนการ Incident Management ให้รองรับการตอบสนองภายใน 4 ชั่วโมง (เช่น การตั้งค่า Alert, การจัดสรรทีมสนับสนุน, การใช้ Automation Scripts) 3. จัดทำรายงานการทดสอบ DR (Disaster Recovery) อย่างสม่ำเสมอเพื่อยืนยันว่าระบบสามารถฟื้นฟูได้ภายใน RTO ที่กำหนด 4. สื่อสารให้ลูกค้าทราบถึงความสามารถและข้อจำกัดของระบบอย่างโปร่งใส รวมถึงขั้นตอนการขอเปลี่ยนแปลง RTO หากต้องการปรับให้สอดคล้องกับความต้องการของลูกค้า 5. พิจารณาเพิ่มเงื่อนไขการชดเชยหรือค่าปรับในสัญญาเพื่อคุ้มครองความเสี่ยงต่อการไม่สามารถทำตาม RTO ได้MEDIUM
การปรับเปลี่ยน RTO จาก 1 วันเป็น 4 ชั่วโมง (พร้อมข้อสังเกตว่าต้องการเปลี่ยนเป็น 1 วัน) ทำให้ข้อตกลงด้าน SLA มีความคลุมเครือ ซึ่งอาจนำไปสู่การตีความที่แตกต่างระหว่างผู้ให้บริการและลูกค้า ส่งผลให้ความเสี่ยงด้านกฎหมายเพิ่มขึ้นเนื่องจากอาจเกิดการละเมิดสัญญา หากไม่มีการกำหนดและบันทึกข้อตกลงใหม่อย่างชัดเจน นอกจากนี้ การตั้งเป้าหมายฟื้นฟูภายใน 4 ชั่วโมงอาจต้องใช้ทรัพยากรและค่าใช้จ่ายเพิ่มเติมจากที่คาดการณ์ไว้ ทำให้ความเสี่ยงด้านการเงินเพิ่มขึ้นในระดับปานกลาง ส่วนด้านการดำเนินงาน ความท้าทายในการตอบสนองต่อเหตุการณ์วิกฤตภายในระยะเวลาที่สั้นกว่าอาจทำให้ระบบฟื้นฟูไม่ทันเวลา ส่งผลให้การให้บริการหยุดชะงักนานขึ้นและเพิ่มความเสี่ยงต่อการสูญเสียข้อมูลหรือความพึงพอใจของลูกค้า จึงถือว่ามีความเสี่ยงด้านการดำเนินงานอยู่ในระดับสูง		page 5MODIFIED
